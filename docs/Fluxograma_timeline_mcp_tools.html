Fase 0 — Descoberta & Baseline

● Criticidade Baixa Nenhum código é alterado · Leitura pura
discover_project
Agente Claude
Escaneia a aplicação-alvo sem alterar nada. Detecta stack, versão do JDK, sistema de build, APIs deprecadas e gera discovery-report.json com inventário completo de riscos. Internamente executa jdeprscan e jdeps.
runMigrationAudit ⚙ interno · step 8b
Disparado por discover_project · Fase 0
Chamada interna do discover_project, executada como step 8b logo após gravar o discovery-report.json. Roda os mesmos 25 critérios estáticos sobre o estado atual da aplicação — antes de qualquer transformação — e gera audit-report-phase-0.md em .jdk-migration/. Este arquivo é o checklist baseline: registra o ponto de partida da migração e serve de referência de comparação com o audit-report-phase-5.md gerado ao final. Não bloqueia a descoberta em caso de erro.
check_internal_dependencies
Agente Claude
Consulta o Nexus / Artifactory corporativo para verificar se as bibliotecas internas (não-públicas) já possuem versões compatíveis com Spring Boot 3 e Jakarta. Evita surpresas tardias de dependência proprietária.
build_migration_plan
Agente Claude
Consolida o relatório de descoberta em um plano faseado revisável com estimativas de ROI, gates por fase e recipes a aplicar. Gera migration-plan.json. É o documento que o humano vai aprovar antes de qualquer transformação.
get_phase_status
Agente Claude · qualquer momento
Retorna o estado atual de todas as fases: pending, in_progress, done, failed. Pode ser chamada a qualquer momento para inspecionar o progresso. Útil para resumir situação ao usuário.
Gate 0 · Humano aprova plano e confirma cobertura de testes
request_gate_approval
MCP automático
Gera um PIN de 6 dígitos que expira em 30 minutos e exibe ao responsável humano. Coleta também as pendingHumanDecisions — decisões que precisam de aprovação antes de avançar de fase. O PIN fica em .gate-pins.json (inacessível ao agente).
approve_gate
Humano digita o PIN
Recebe o PIN digitado explicitamente pelo humano, valida e emite um token que desbloqueia a fase seguinte. Rejeita nomes como "bot", "claude" ou "agent" no campo approverName. Só o humano pode desbloquear.
generate_report
MCP automático · após cada gate
Gera o relatório HTML de auditoria (audit-report-{ts}.html) com trilha completa de decisões, tokens, ROI e diff de cada fase aprovada. Chamado automaticamente após cada approve_gate e pode ser chamado manualmente a qualquer momento.
Nenhum arquivo da aplicação é modificado nesta fase. Toda saída é escrita em .jdk-migration/ na raiz do projeto-alvo.
1

Fase 1 — Infraestrutura & Build

● Criticidade Baixa POM/Gradle, Toolchain JDK 21, Wrapper · Gate: build verde no JDK 21
execute_phase
Agente Claude
Aplica a fase usando o token do gate aprovado. Cria branch Git isolada (jdk-migration/phase-1-…), executa os recipes OpenRewrite/SBM/ET definidos no plano, faz commit e retorna diff. Suporta dryRun: true para simulação sem commit.
update_step_status
MCP automático · durante a fase
Registra o progresso granular de cada step dentro da fase em execução: in_progress → done / failed. Em execução manual (execute_phase falhou), deve ser chamada imediatamente após cada step — nunca retroativamente.
record_manual_phase
Agente Claude · se execute_phase falhar
Alternativa quando execute_phase falha por problema ambiental (ex: spawn EINVAL no Windows). O agente executa os steps manualmente e chama esta tool para registrar a fase no estado do MCP, garantindo trilha de auditoria e possibilidade de rollback.
rollback_phase
Agente Claude · se build falhar
Reverte a fase aplicada via Git para o estado anterior. Só é chamado quando a fase produziu um estado inconsistente e o build quebrou irrecuperavelmente. Cada fase em branch isolada torna o rollback cirúrgico e seguro.
update_phase_costs
MCP automático · ao concluir fase
Atualiza os tokens reais consumidos na fase e recalcula o ROI: custo real de IA versus esforço humano estimado poupado. Alimenta o dashboard de ROI no relatório final. Se tokenUsage for omitido, recalcula via JSONL de sessão.
Gate 1 · Build verde no JDK 21 · request → PIN → approve
request_gate_approval
MCP automático
Gera o PIN e apresenta as decisões pendentes antes de liberar a Fase 2.
approve_gate
Humano digita o PIN
Valida o PIN e emite token que libera a Fase 2. Também dispara generate_report automaticamente.
2

Fase 2 — Modernização de Linguagem

● Criticidade Média Records, Sealed Classes, var, switch expressions · Gate: diff revisado + testes verdes
execute_phase
Agente Claude
Aplica recipes de modernização de linguagem Java: var, records, sealed classes, switch expressions, text blocks. Exige dryRun obrigatório antes de stacks de média/alta complexidade.
update_step_status
MCP automático
Registra progresso granular de cada transformação de linguagem aplicada.
update_phase_costs
MCP automático
Recalcula ROI com tokens reais consumidos nesta fase de modernização.
Gate 2 · Diff revisado + testes verdes
request_gate_approval
MCP automático
Gera PIN e coleta decisões pendentes (ex: revisar diff manualmente antes de aprovar).
approve_gate
Humano digita o PIN
Valida PIN e libera Fase 3. Relatório de auditoria gerado automaticamente.
3

Fase 3 — Namespace Jakarta & Frameworks

● Criticidade Alta javax → jakarta, Spring Boot 3, Spring Batch 5 · Gate: app sobe + smoke tests
execute_phase
Agente Claude
A fase mais densa mecanicamente: converte javax.* → jakarta.*, migra Spring Boot 2→3, Spring Batch 4→5 e APIs Servlet. Usa Eclipse Transformer para JARs sem equivalente jakarta. dryRun obrigatório antes de executar.
update_step_status
MCP automático
Registra progresso de cada conversão de namespace e migração de framework.
rollback_phase
Agente Claude · se app não subir
Se a app não subir após as transformações de namespace, reverte tudo via Git de forma cirúrgica — sem afetar outras fases.
update_phase_costs
MCP automático
Registra custo real da fase de maior consumo de tokens do ciclo.
Gate 3 · App sobe no servidor + smoke tests passam
request_gate_approval
MCP automático
Gera PIN e apresenta evidências de que a app subiu com sucesso.
approve_gate
Humano digita o PIN
Aprovação crítica — confirma que a app funciona no novo runtime antes da refatoração semântica.
4

Fase 4 — Refatoração Semântica Assistida

● Criticidade CRÍTICA EJB, JTA, JSF, PrimeFaces · Gate: paridade funcional item a item
execute_phase
Agente Claude
Para stacks EJB/JTA: apenas sinaliza e organiza a revisão humana — nunca aplica transformações semânticas automaticamente. Para JSF: converte namespace automaticamente, mas UI é validada manualmente. dryRun obrigatório.
record_manual_phase
Agente Claude · EJB/JSF manuais
Fase com maior uso de execução manual. Após o humano concluir refatorações de EJB/JSF, o agente registra cada step no MCP para manter rastreabilidade e possibilidade de rollback.
update_step_status
MCP automático
Registra progresso de cada item da checklist de paridade funcional.
update_phase_costs
MCP automático
Fase mais cara em esforço humano — ROI calculado reflete o delta de automação assistida vs. puramente manual.
get_phase_status
Agente Claude · sob demanda
Consultado frequentemente nesta fase longa para acompanhar quais items da checklist de paridade já foram concluídos.
Gate 4 · Paridade funcional verificada item a item
request_gate_approval
MCP automático
Gera PIN com checklist de paridade funcional completa para revisão humana.
approve_gate
Humano digita o PIN
Gate mais rigoroso do ciclo — confirma paridade funcional completa antes do cutover final.
5

Fase 5 — Validação Final & Cutover

● Criticidade Média Performance, segurança, cutover · Gate: sign-off liderança técnica e funcional
execute_phase
Agente Claude
Executa testes finais de performance, segurança e regressão. Consolida evidências para o sign-off. Gera o checklist de resultado pós-migração (audit-report-phase-5.md).
get_phase_status
Agente Claude
Verifica que todas as 5 fases anteriores estão com status done antes de apresentar ao responsável técnico.
update_phase_costs
MCP automático
Consolida custo total do projeto e ROI acumulado de todas as fases para o relatório final.
Gate 5 · Sign-off liderança técnica e funcional → CUTOVER
request_gate_approval
MCP automático
Gera o PIN final. Apresenta sumário completo: ROI total, todas as fases concluídas, evidências de paridade. Requer dois approverNames distintos (tech lead + funcional).
approve_gate
Humano · sign-off final
Aprovação final. Marca a migração como concluída. Desbloqueia o cutover para produção e aciona a geração do relatório final definitivo.
runMigrationAudit ⚙ interno
Disparado por generate_report · Fase 5
Função interna de src/static-analysis/migration-audit.ts — não é uma MCP tool, mas é o coração da análise final. Executada automaticamente por generate_report quando a Fase 5 está in_progress, awaiting_gate, approved ou completed. Roda 25 critérios estáticos sobre o projeto-alvo e retorna um MigrationAuditResult com status ok / warning / fail por critério, além de um veredicto fundamentado (approved · conditionally_approved · rejected) que aparece como banner no relatório HTML.

generate_report
MCP automático · relatório FINAL
Gera o audit-report-final.html — relatório consolidado com trilha completa de auditoria, ROI de todas as fases, diff acumulado e timeline de gates. Na Fase 5, incorpora automaticamente o resultado dos 25 critérios do runMigrationAudit (chamada acima) e o veredicto fundamentado (aprovado / aprovado condicionalmente / rejeitado) como banner de destaque no topo do documento. O artefato definitivo da migração concluída.
