2

Configurar o MCP no projeto-alvo

Um único arquivo, um único bloco. Faça exatamente isto.

💡 Leia primeiro — por que antes era confuso: não existem "duas etapas" (instalar + configurar). O download do pacote é feito automaticamente pelo npx. Você só precisa de UM arquivo de configuração dizendo ao Claude Code como iniciar o servidor. Esqueça npm install manual e múltiplos arquivos. Siga os 3 passos abaixo.

2.1
Passo 1 — Abra (ou crie) o arquivo ~/.claude.json
Este é o arquivo de configuração global do Claude Code. No Windows fica em C:\Users\SEU_USUARIO\.claude.json. É o único arquivo que você vai editar.
📄 C:\Users\SEU_USUARIO\.claude.json👤 Humano

Por que este arquivo e não .claude/settings.json do projeto? Porque no Windows o processo do MCP não herda o PATH do seu terminal — e o ~/.claude.json permite declarar o bloco env com PATH e variáveis de JDK que o servidor realmente vai enxergar. É a forma mais robusta e a que evita 90% dos erros de configuração.

2.2
Passo 2 — Cole o bloco do servidor jdk-migration
Adicione (ou complete) a chave mcpServers com o bloco abaixo. Ajuste os caminhos para os da sua máquina (anotados na Seção 1).
📄 ~/.claude.json👤 Humano
~/.claude.json — copie e ajuste os caminhos
{
  "mcpServers": {
    "jdk-migration": {
      "command": "npx",
      "args": ["-y", "jdk-migration-mcp"],
      "env": {
        // PATH: o MCP precisa achar npx (node), mvn, java e git
        "PATH": "C:\\Program Files\\nodejs;C:\\Users\\SEU_USUARIO\\AppData\\Roaming\\npm;C:\\devtools\\Apache\\maven\\bin;C:\\Program Files\\Zulu\\zulu-21\\bin;C:\\Program Files\\Git\\cmd;C:\\WINDOWS\\system32",
        "PATHEXT": ".COM;.EXE;.BAT;.CMD;.VBS;.WSH;.MSC",
        // JDK 21 (destino) — usado nas Fases 1–5
        "JAVA_HOME": "C:\\Program Files\\Zulu\\zulu-21",
        // JDK de ORIGEM (ex: 8) — usado na Fase 0 para compilar o projeto legado
        "SOURCE_JAVA_HOME": "C:\\Program Files\\Zulu\\zulu-8",
        "MAVEN_HOME": "C:\\devtools\\Apache\\maven"
      }
    }
  }
}
✔ Anatomia do bloco — o que cada campo faz
  • command + args: manda o Claude Code rodar npx -y jdk-migration-mcp — o npx baixa e executa a versão mais recente sozinho. Nada para instalar manualmente.
  • env.PATH: o campo mais importante. Sem nodejs aqui, o npx nem inicia. Sem o bin do Maven/JDK, o build falha com spawn ENOENT.
  • env.JAVA_HOME: aponta para o JDK 21 (destino).
  • env.SOURCE_JAVA_HOME: aponta para o JDK de origem (ex: 8). É o fallback que a Fase 0 usa para compilar o projeto legado. essencial v4
  • env.MAVEN_HOME: raiz do Maven (o bin já está no PATH).
⚠ Regras do JSON no Windows
  • Use sempre barras duplas \\ nos caminhos.
  • Coloque C:\Program Files\nodejs no início do PATH.
  • Troque SEU_USUARIO pelo seu usuário real do Windows.

7

JDK na Fase 0 — em detalhe

Como JAVA_HOME, sourceJdkHome e targetJdkHome são atribuídos durante o discover_project

O processo MCP roda com JAVA_HOME apontando para o JDK 21. Mas para analisar o projeto legado, a Fase 0 precisa compilá-lo com o JDK de origem. Por isso o MCP separa três conceitos:

🔵 JAVA_HOME

C:\...\zulu-21

JDK 21 do processo MCP. É o ambiente em que o servidor Node roda. Nunca é alterado durante a migração.

🟠 sourceJdkHome

C:\...\zulu-8

JDK de origem do projeto (6 ou 8). Nunca pode ser 21. Usado para compilar o legado na Fase 0 antes do jdeprscan.

🔵 targetJdkHome

C:\...\zulu-21

JDK de destino — sempre 21. Usado nas Fases 1–5 para compilar e testar a aplicação modernizada.

7.1
Como o MCP decide o sourceJdkHome (origem)
A busca casa com a versão do projeto (ex: 8). Repare que o JAVA_HOME não é usado aqui — para não pegar o JDK 21 por engano.
Ordem de busca do JDK de origem (ex: projeto JDK 8)
1. toolOverrides do usuário ......... SOURCE_JAVA_HOME / JAVA_HOME_8
2. variáveis de ambiente ............ SOURCE_JAVA_HOME, JDK_8, JAVA_HOME_1_8
3. paths comuns de instaladores ..... zulu-8, temurin-8, jdk1.8.0, /usr/lib/jvm/...
4. valida a versão .................. precisa ser 8 (ou 6); se for 21 → REJEITA
5. nada válido encontrado ........... PEDE o caminho ao usuário
🟠 Regra inviolável

Um candidato com versão 21 nunca é aceito como origem. Se o seu JAVA_HOME aponta para 21 e não há SOURCE_JAVA_HOME, o MCP ignora o 21 e segue procurando o JDK 8 — ou pede a você.

7.2
⚠ E quando o JDK de origem NÃO é encontrado?
Cenário comum: projeto é JDK 8, mas você não configurou SOURCE_JAVA_HOME e o JDK 8 não está em um path conhecido. Veja o que acontece.
O MCP interrompe e devolve uma mensagem clara
# Resposta do discover_project quando falta o JDK de origem:
{
  "status": "missing_tools",
  "message": "JDK 8 não encontrado automaticamente. O projeto-alvo usa
              JDK 8 — informe o path via toolOverrides.
              Chave: SOURCE_JAVA_HOME"
}
Você então informa o caminho e o Claude reexecuta a descoberta:
Sua resposta ao Claude Code
Execute discover_project no projeto C:\meu-projeto

✔ Como evitar essa parada
  • Deixe SOURCE_JAVA_HOME configurado no env de ~/.claude.json desde o início (Seção 2.2).
  • Assim a etapa 2 da busca já encontra o JDK e a descoberta roda sem interrupção.

Aprovação de Gate com PIN

A barreira técnica que impede aprovação autônoma pelo agente

10.1
Sequência passo a passo
Como um gate é aprovado
── CLAUDE (após execute_phase) ─────────────────────────────
1. request_gate_approval(projectPath, faseN)
   → PIN gerado, salvo em disco, exibido na tela:

   ══════════════════════════════════════════
     PIN DE APROVAÇÃO — FASE 2  ➜  4 8 2 9 1 3
   ══════════════════════════════════════════

── VOCÊ (humano) ───────────────────────────────────────────
2. Revisa diff / build / testes
3. Lê o PIN e responde:
   "Aprovado. PIN: 482913. Aprovador: Matheus Castro"

── CLAUDE (com o PIN) ──────────────────────────────────────
4. approve_gate(projectPath, 2, "Matheus Castro", "482913")
   → valida PIN (uso único) → emite gateToken da Fase 3
🔐 Exemplo de PIN exibido
4 8 2 9 1 3

Aparece no terminal do Claude Code. Informe ao agente junto com o "aprovado". Expira em 30 minutos.

🔐 Por que a IA não consegue burlar
  • PIN aleatório (900.000 combinações), salvo em .jdk-migration/.gate-pins.json — nenhuma tool retorna esse arquivo
  • Uso único + expira em 30 min
  • Nomes de sistema bloqueados: bot, claude, ai, agent, ci, gpt…
  • Token final assinado por HMAC-SHA256 derivado do caminho do projeto, validade 30 dias

Troubleshooting

Problemas comuns e solução direta

T0
MCP não conecta — npx não encontrado
✔ Solução
  • Adicione C:\Program Files\nodejs no início do env.PATH em ~/.claude.json (Seção 2.2).
  • Inclua ...\AppData\Roaming\npm. Reinicie o Claude Code.
T1
spawn ENOENT — mvn / java / git não encontrado
✔ Solução
  • Adicione o bin do Maven e do JDK ao env.PATH e inclua PATHEXT (Seção 2.2).
T2
JDK de origem não encontrado na Fase 0
✔ Solução
  • Defina SOURCE_JAVA_HOME no env de ~/.claude.json (Seção 2.2), ou
  • Informe na hora via toolOverrides: { "SOURCE_JAVA_HOME": "C:\\...\\zulu-8" } (Seção 7.2).
T3
MCP pegou o JDK errado como origem (pegou o 21)

Não ocorre: o detector rejeita qualquer candidato versão 21 como origem. Se a origem ficou vazia, é porque o JDK 8/6 não foi localizado — informe SOURCE_JAVA_HOME (T2). Confira o valor gravado em jdk-migration.config.json → sourceJdkHome.

T4
GATE_TOKEN_INVALID — PIN incorreto/expirado
✔ Solução
  • PIN expira em 30 min — peça ao Claude para chamar request_gate_approval de novo.
  • Use seu nome real em approverName (não "bot"/"claude"/"ai").
T5
Fase presa em failed

execute_phase reseta failed → pending automaticamente na nova tentativa. Se o trabalho foi feito na mão, registre com record_manual_phase (em branch isolada criada com git checkout -b jdk-migration/phase-N-...).
