Este guia cobre o processo completo de instrumentação de um projeto Java legado — desenvolvido por humanos sem apoio de IA — para que passe a evoluir com o Claude Code. O objetivo é fazer com que o Claude tenha contexto suficiente para trabalhar com qualidade e segurança em um codebase que ele não conhece.

O processo está dividido em 3 fases: Descoberta, Instrumentação e Workflow. Siga em ordem na primeira vez; depois, consulte conforme necessário.

00
Fase 0
Premissas
Prepare o repositório em uma branch isolada antes de qualquer trabalho de migração.

Antes de iniciar qualquer etapa de migração, é fundamental garantir que todo o trabalho seja realizado em uma branch isolada no GitHub. Isso protege a branch RELEASE e garante que o merge final seja feito exclusivamente por um humano (gate humano).

1
Acesse o seu workspace local
Navegue até a pasta onde se localizam os seus projetos em sua máquina local.
2
Localize o repositório no GitHub
Acesse o GitHub, selecione o repositório que será migrado e selecione também a branch RELEASE.
3
Clone o repositório localmente
Execute o comando abaixo para gerar uma cópia em seu workspace:
bash — workspace
$ git clone <url do repositório>
Cloning into 'meu-projeto'...
remote: Enumerating objects: 1234, done.
Receiving objects: 100% (1234/1234), done.
4
Crie uma branch de migração isolada no GitHub
Desconecte o projeto clonado localmente e peça para criar uma nova branch no GitHub com o nome migrate/jdk8-to-jdk21-yyyymmdd (substituindo yyyymmdd pela data atual).
5
Commit e push na nova branch
Faça o commit e push do projeto nesta nova branch criada para que ela exista no repositório remoto antes de iniciar o trabalho.
Gate Humano

Realizados os passos acima, todo o trabalho de migração do projeto estará sendo tratado em uma branch isolada. Ao final, após validado, os PRs desta branch para a branch RELEASE devem ser executados exclusivamente por um humano — nunca por automação.

Importante

Se você estiver executando o MCP fora da Cielo — é necessário estar conectado à VPN Cielo para que a ferramenta funcione corretamente. Verifique sua conexão antes de prosseguir.

Fase 1
Descoberta
Deixe o Claude entender o projeto antes de qualquer coisa.

Antes de criar qualquer arquivo de configuração, o Claude precisa explorar o codebase. Navegue até a raiz do projeto e abra o Claude Code:

PowerShell / Git Bash
PS C:\> cd C:\caminho\para\seu-projeto-java
PS projeto> claude
✻ Welcome to Claude Code!
 
claude> /init
→ Scanning project structure...
→ Reading pom.xml and dependencies...
→ Generating CLAUDE.md...
Nota
O /init funciona também em projetos existentes — o Claude revisa a estrutura e sugere melhorias baseado no que encontra no codebase. Ele gera um rascunho do CLAUDE.md que você vai revisar e enriquecer.

Rodada de perguntas exploratórias

Depois do /init, antes de editar qualquer coisa, execute esta rodada de perguntas para validar o que o Claude entendeu:

Sessão exploratória
claude> Descreva a arquitetura geral deste projeto
claude> Quais são os principais módulos e responsabilidades de cada um?
claude> Quais padrões de design estão sendo usados?
claude> Quais são as dependências externas e integrações?
claude> Onde estão os pontos de entrada da API (controllers/endpoints)?
claude> Existe alguma convenção de nomenclatura ou estrutura de pacotes?
claude> Que frameworks e versões estão sendo usados? (leia o pom.xml)

Use as respostas para alimentar e corrigir o CLAUDE.md que o /init gerou. Qualquer ponto onde o Claude errar ou mostrar lacuna é um candidato a seção no arquivo.

Pré-requisito
Se o projeto ainda não tem Git inicializado, faça isso antes de qualquer coisa. O Claude Code trabalha com diff e histórico para entender mudanças e criar guardrails seguros. 
git init
git add .
git commit -m "chore: initial commit before Claude Code setup"
02
Fase 2
Instrumentação
Os arquivos que você precisa criar para dar contexto permanente ao Claude.

A instrumentação consiste em criar um conjunto de arquivos que dão ao Claude contexto persistente sobre o projeto — convenções, restrições, comandos, estrutura e guardrails. Esses arquivos ficam no repositório e são lidos em toda sessão.

seu-projeto-java/
├── CLAUDE.md ← cérebro do projeto (commitar)
├── .claude/
│ ├── settings.json ← permissões e guardrails (commitar)
│ ├── settings.local.json ← config pessoal (.gitignore)
│ └── commands/ ← slash commands customizados (commitar)
│ ├── nova-feature.md
│ ├── revisar-pr.md
│ └── adicionar-teste.md
├── pom.xml
└── src/

2.1 — CLAUDE.md O mais importante

Este arquivo é lido pelo Claude no início de toda sessão. Para um projeto legado, ele precisa ser especialmente rico — vai substituir anos de contexto que humanos acumularam organicamente sobre o codebase.

# [Nome do Projeto]

## Visão Geral
[O que faz, qual domínio de negócio, quem usa]

## Stack Técnica
- Java [versão] (Eclipse Temurin)
- [Spring Boot X.Y / Jakarta EE / Quarkus / etc.]
- Maven — pom.xml na raiz
- [JPA/Hibernate / MyBatis / JDBC puro]
- [Banco: Oracle 19c / PostgreSQL 14 / etc.]
- [Autenticação: JWT / OAuth2 / LDAP / etc.]
- [Servidor: Tomcat embutido / JBoss / WebLogic / etc.]

## Estrutura de Pacotes
```
com.empresa.projeto
├── controller/     # endpoints REST
├── service/        # lógica de negócio
├── repository/     # acesso a dados
├── domain/         # entidades e DTOs
├── config/         # beans de configuração
└── util/           # utilitários
```

## Comandos Essenciais
```bash
mvn clean install          # build completo
mvn test                   # testes unitários
mvn verify                 # build + integração
mvn spring-boot:run        # subir local
```

## Convenções do Projeto (IMPORTANTE)
- Controllers nomeados como XxxController
- Padrão de resposta: ResponseEntity<ApiResponse<T>>
- Exceções: GlobalExceptionHandler via @ControllerAdvice
- @Transactional sempre na service, nunca no controller
- Logs: SLF4J com Lombok @Slf4j

## Áreas Críticas — Alterar com Cuidado
- XxxLegacyService — sem testes, risco alto de regressão
- db/migrations/ — nunca editar existentes, apenas criar novas
- XxxIntegration — integração com sistema externo de produção

## Decisões Arquiteturais
- "Usamos JDBC puro no módulo de relatórios por performance — não migrar para JPA"
- "DTOs mapeados manualmente, não usamos MapStruct por decisão do time"

## O Que NÃO Fazer
- Não usar `var` — mantemos compatibilidade de leitura com Java 8
- Não adicionar dependências ao pom.xml sem discussão
- Não refatorar código legado sem cobertura de testes primeiro

## Contexto de Testes
- Framework: JUnit 5 + Mockito
- Espelha estrutura do main em src/test/java
- Cobertura atual: [X%] — módulos sem cobertura: [listar]
Regra de Ouro
Mantenha o CLAUDE.md abaixo de 200 linhas. Priorize documentar o que o Claude costuma errar sobre convenções gerais. Qualquer erro de convenção que ele cometer durante o desenvolvimento — adicione imediatamente uma nota em "O Que NÃO Fazer".

2.2 — CLAUDE.md por módulo/pacote

Para projetos grandes com múltiplos módulos Maven ou pacotes com lógica muito específica, crie CLAUDE.md dentro das subpastas relevantes:

src/main/java/com/empresa/projeto/
├── CLAUDE.md ← regras gerais de código Java
├── pagamento/
│ └── CLAUDE.md ← regras específicas do módulo
└── relatorios/
└── CLAUDE.md ← "use JDBC puro, não JPA"

O Claude lê automaticamente o CLAUDE.md mais próximo de onde está trabalhando — útil para isolar regras que só se aplicam a um contexto específico.

2.3 — .claude/settings.json Guardrails

Define o que o Claude tem permissão de ler, escrever e executar. Para um projeto legado, isso é essencial — você quer guardrails que impeçam alterações acidentais em áreas críticas.

{
  "permissions": {
    "allow": [
      "Read(**)",
      "Write(src/main/java/**)",
      "Write(src/test/java/**)",
      "Write(src/main/resources/**)",
      "Bash(mvn clean install)",
      "Bash(mvn test)",
      "Bash(mvn verify)",
      "Bash(mvn spring-boot:run)"
    ],
    "deny": [
      "Write(pom.xml)",
      "Write(db/migrations/**)",
      "Write(.env)",
      "Write(*.properties)",
      "Bash(git push*)"
    ]
  }
}
Como funciona
Ações dentro de allow são executadas diretamente. Ações em deny são bloqueadas — o Claude pedirá sua confirmação manual. Adapte os bloqueios conforme o que for crítico no seu projeto.

2.4 — Slash commands customizados

Crie a pasta .claude/commands/ com comandos que você usará com frequência. Cada arquivo .md vira um comando acessível com /nome-do-arquivo dentro do Claude Code.

.claude/commands/nova-feature.md

Implemente uma nova feature seguindo o padrão do projeto:

1. Leia o CLAUDE.md para relembrar as convenções
2. Crie o Controller em `controller/` com anotações padrão
3. Crie a Service em `service/` com @Service e @Transactional
4. Crie o Repository em `repository/`
5. Crie os DTOs necessários em `domain/dto/`
6. Escreva testes unitários para a Service
7. Descreva o que foi feito e o que precisa de atenção

Feature a implementar: $ARGUMENTS

.claude/commands/revisar-pr.md

Revise o código modificado nesta sessão:

1. Verifique aderência às convenções do CLAUDE.md
2. Identifique riscos de regressão
3. Verifique se há testes cobrindo os caminhos críticos
4. Aponte código legado tocado sem cobertura de testes
5. Sugira melhorias sem implementar — apenas liste

.claude/commands/adicionar-teste.md

Adicione testes para: $ARGUMENTS

Siga o padrão do projeto:
- JUnit 5 + Mockito
- Padrão AAA (Arrange, Act, Assert) com comentários
- Nomes descritivos: `deveRetornarErroQuando[Cenario]()`
- Mocke apenas dependências externas, não a classe sob teste
Como usar
Dentro do Claude Code, digite /nova-feature endpoint de cancelamento de pedido — o $ARGUMENTS será substituído pelo texto que você passar.

2.5 — Atualizar o .gitignore

Adicione ao .gitignore existente do projeto:

# Claude Code — configurações pessoais (não commitar)
.claude/settings.local.json

# Memória automática gerada por sessão (é local)
.claude/memory/

Regra de git para o time: commite CLAUDE.md, .claude/settings.json e .claude/commands/ para que todo o time compartilhe o mesmo contexto. Mantenha settings.local.json fora do git — ele guarda preferências pessoais de cada desenvolvedor.

03
Fase 3
Workflow
Como trabalhar no dia a dia com Claude Code em um projeto legado.

O ciclo correto para projetos legados

1
Abrir Claude Code na raiz
cd seu-projeto && claude — sempre a partir da raiz para o CLAUDE.md ser lido.
2
Dar contexto da tarefa
Descreva o objetivo em linguagem natural. Quanto mais contexto de negócio, melhor.
3
Pedir exploração ANTES da implementação
Para tarefas complexas, peça que o Claude leia os arquivos relevantes e explique o fluxo atual antes de propor qualquer mudança.
4
Usar Plan Mode — sempre para mudanças estruturais
Pressione Shift + Tab para ativar. O Claude descreve o plano completo e aguarda sua aprovação antes de tocar em qualquer arquivo.
5
Aprovar → Claude implementa
Após revisar e aprovar o plano, o Claude executa as mudanças seguindo exatamente o que foi combinado.
6
Revisar o diff com atenção
Leia cada alteração. Para legado, preste atenção especial em mudanças que tocam código sem cobertura de testes.
7
Rodar os testes
mvn test ou mvn verify — nunca commite sem a suíte de testes passar.
8
Commitar
Commits atômicos e descritivos. O histórico de git é contexto valioso para futuras sessões do Claude.

Prompt de abertura para tarefas complexas

Para tarefas que tocam lógica de negócio crítica, use este padrão antes de pedir código:

Prompt recomendado para legado
claude> Antes de implementar, leia os arquivos:
  - src/main/java/com/empresa/projeto/[Módulo]Service.java
  - src/main/java/com/empresa/projeto/[Módulo]Controller.java
  - CLAUDE.md
 
  Me explique como o fluxo de [funcionalidade] funciona hoje,
  e só depois me diga sua proposta de implementação.
  Aguarde minha aprovação antes de alterar qualquer arquivo.

Plan Mode — atalho essencial

Ativar / Desativar
Shift + Tab
Alterna entre modo de execução e modo de planejamento dentro de uma sessão ativa.
Por que usar
Para mudanças estruturais
O Claude descreve o que vai fazer e por quê antes de tocar em qualquer arquivo. Você pega premissas erradas antes que virem código.

Checklist Final de Instrumentação

Marque mentalmente cada item antes de considerar o projeto instrumentado:

  • Git inicializado e projeto com commit inicial criado
  • /init executado — CLAUDE.md rascunho gerado
  • Rodada de perguntas exploratórias concluída
  • CLAUDE.md revisado e enriquecido com contexto real do projeto
  • CLAUDE.md por subpasta para módulos críticos (se necessário)
  • .claude/settings.json criado com permissões e guardrails corretos
  • .claude/commands/ com slash commands do dia a dia
  • .gitignore atualizado com entradas do Claude Code
  • CLAUDE.md e .claude/settings.json commitados no git
  • Primeiro ciclo completo validado: Plan → Implementa → mvn test → Commit

O investimento de 1 a 2 horas nessa instrumentação inicial vai te poupar de reexplicar contexto em cada sessão e vai evitar que o Claude "invente" convenções que contradizem o projeto legado. Qualquer erro de convenção que ele cometer — adicione imediatamente ao CLAUDE.md na seção "O Que NÃO Fazer".
