01 — Contexto e Objetivo

Por que este projeto é crítico.

Estamos propondo uma ferramenta que altera código de aplicações já em ambiente produtivo. O risco não está na troca de um número de versão — está na heterogeneidade das aplicações-alvo e na possibilidade de regressões silenciosas em sistemas que sustentam o negócio.

A migração de uma aplicação do JDK 6 ou 8 para o JDK 21 atravessa de treze a quinze versões intermediárias do Java. Cada uma acumulou mudanças de linguagem, remoções de API, e — sobretudo — três rupturas estruturais: o Sistema de Módulos (JDK 9), a remoção dos módulos Java EE e CORBA (JDK 11) e a encapsulação forte dos internos do JDK (JDK 17). Esta análise está detalhada no documento de avaliação de diferenças entre versões que fundamenta esta proposta.

Premissa central da solução

Não vamos reinventar a transformação de código. A indústria já consolidou ferramentas maduras para isso — notavelmente o OpenRewrite, que possui recipes oficiais mantidos pela própria Oracle para WebLogic + JDK 21, além de recipes da comunidade para Spring Boot e Jakarta EE. Nossa solução orquestra esse ecossistema, adiciona a camada de descoberta e diagnóstico por stack, e impõe a governança de revisão humana que um ambiente produtivo exige.

O que entregamos na v1

  • Dois caminhos de migração: JDK 6 → 21 e JDK 8 → 21, sendo o primeiro um superconjunto de esforço do segundo.
  • Instalável na aplicação-alvo: via servidor MCP registrável no agente de IA (VS Code, Claude Code, Cursor) e uma Skill de instalação que parametriza o comportamento por stack detectada.
  • Diagnóstico antes de qualquer alteração: a ferramenta primeiro entende a aplicação, classifica a stack, e produz um plano de migração faseado e revisável.
  • Governança human-in-the-loop: nenhuma fase avança sem um gate de aprovação explícita.
02 — Princípios de Design

Quatro decisões que
guiam toda a solução.

1. Orquestrar, não reimplementar

Transformação de código em escala é problema resolvido pelo OpenRewrite e ferramentas correlatas. Reescrevê-las seria desperdício e fonte de bugs. Nossa solução é a camada de inteligência, diagnóstico e governança sobre esses motores.

2. Diagnóstico antes de ação

Toda aplicação passa por uma fase de descoberta que identifica build system, stack, versão atual, dependências e pontos de incompatibilidade — produzindo um plano antes de tocar em uma linha de código.

3. Reversibilidade por construção

Cada fase ocorre em branch Git dedicada, com commit isolado e checkpoint. Falha de build dispara rollback automático. Nada é destrutivo e nada é irreversível sem aprovação.

4. O humano é a última linha

A automação cobre o trabalho mecânico. Mudanças semânticas — lógica de transação EJB, beans stateful, segurança — são sempre sinalizadas para revisão humana, nunca aplicadas às cegas.

"A ferramenta acelera o que é seguro automatizar e torna visível o que exige julgamento humano. Ela nunca confunde as duas coisas."

03 — Arquitetura da Solução

Um MCP orquestrador
com uma Skill de instalação.

A solução combina dois componentes: um servidor MCP que expõe as ferramentas de diagnóstico e migração ao agente de IA, e uma Skill de instalação que parametriza o comportamento conforme a stack detectada na aplicação-alvo.

Por que MCP, e não um plugin tradicional

Um servidor MCP é, por design, um pacote distribuível (via npm) que roda como processo local e se conecta a qualquer host compatível — VS Code, Claude Code, Cursor. Isso resolve "instalável na aplicação-alvo" sem amarrar a solução a uma IDE específica, e dá acesso nativo ao sistema de arquivos do projeto, ao Git, e à execução de processos (Maven, Gradle, jdeprscan, OpenRewrite). Um plugin de IDE seria mais limitado e exigiria reimplementação por editor.

A Skill de instalação e parametrização

No momento da instalação na aplicação-alvo, uma Skill conduz a parametrização: detecta a stack (Spring Boot, EJB, JSF, etc.), pergunta o que não puder inferir (servidor de aplicação, versão-alvo de framework, ambiente de CI), e grava um arquivo de configuração do projeto (jdk-migration.config.json) que orienta todas as fases seguintes. É a Skill que adapta a ferramenta genérica ao caso concreto daquela aplicação.

04 — Ferramentas do MCP

Três tools, um fluxo
de confiança progressiva.

A superfície de interação é deliberadamente enxuta. O agente de IA precisa saber apenas quando descobrir, quando planejar e quando executar — sempre mediado por um gate humano.

Tool 01

discover_project

Escaneia a aplicação sem alterá-la. Identifica build system, stack, versão de JDK atual, dependências e pontos de incompatibilidade.

  • Detecta Maven/Gradle/Ant e multi-módulo
  • Classifica a stack via Stack Profilers
  • Roda jdeprscan + jdeps contra a versão alvo
  • Produz inventário de riscos por categoria
Tool 02

build_migration_plan

Consolida o diagnóstico em um plano faseado, classificado por criticidade, com gates definidos. Gera o documento de proposta de migração.

  • Ordena fases da menos para a mais crítica
  • Mapeia recipes OpenRewrite aplicáveis por fase
  • Estima esforço e sinaliza revisões manuais
  • Define o gate e o critério de aprovação de cada fase
Tool 03

execute_phase

Aplica uma única fase aprovada. Exige o token do gate. Cada fase é uma branch com commit isolado e build de validação.

  • Verifica token de aprovação do gate anterior
  • Suporta dryRun para preview do diff
  • Roda build + testes; rollback automático em falha
  • Abre Pull Request com relatório da fase
Aux

auxiliares

Ferramentas de apoio à governança e recuperação, usadas pelo orquestrador e pelo revisor humano.

  • get_phase_status — situação de cada fase
  • approve_gate — registra aprovação humana
  • rollback_phase — reverte uma fase aplicada
  • generate_report — relatório consolidado

05 — Desafios por Stack

Cada arquitetura tem
seus próprios abismos.

A diversidade das aplicações-alvo é o maior fator de risco. Um microservice Spring Boot stateless é quase trivial; uma aplicação JSF com EJB rodando em WebLogic é um projeto à parte. Os Stack Profilers existem exatamente para tratar cada caso com o conhecimento específico que ele exige.

Microservices REST / stateless

COMPLEXIDADE BAIXA

Sem estado, com testes e CI maduros. Migração majoritariamente automatizável via recipe de JDK do OpenRewrite. Caso ideal para iniciar o projeto e validar a ferramenta.

Spring Boot 2.x

COMPLEXIDADE ALTA

Exige salto para Spring Boot 3 (que requer JDK 17+), trazendo a conversão obrigatória javax → jakarta, mudanças no Spring Security 6 e no RestTemplate. Spring Boot Migrator automatiza o mecânico; segurança exige revisão.

Spring Batch

COMPLEXIDADE ALTA

Spring Boot 3 traz o Spring Batch 5, com novo modelo de configuração de jobs. @EnableBatchProcessing legado e beans como JobRepository seguem novos padrões. Lógica de chunk e restart precisa de validação funcional.

EJB

COMPLEXIDADE CRÍTICA

Beans stateful, transações container-managed (JTA), chamadas remotas RMI/IIOP e dependências de JNDI não têm tradução automática segura. Exige redesenho assistido e validação de paridade funcional rigorosa.

JSF / PrimeFaces

COMPLEXIDADE CRÍTICA

Migração de namespace nas XHTML, managed beans de javax.faces.bean para CDI, PrimeFaces com classifier jakarta, e regras de client-ID que mudaram no Faces 4. Nem tudo tem recipe; há trabalho manual de UI.

WebLogic

COMPLEXIDADE ALTA

A Oracle fornece recipes OpenRewrite oficiais (rewrite-weblogic) para migrar para WebLogic 14.1.2 + JDK 21. A regra é clara: rodar o recipe de JDK primeiro, depois o de WebLogic. Conectores JCA e realms de segurança proprietários exigem atenção.

Limite honesto de automação

Ferramentas como OpenRewrite e Spring Boot Migrator resolvem com excelência o trabalho mecânico: troca de namespace, bump de versões, substituição de APIs com mapeamento claro. Elas não resolvem mudanças semânticas — redesenho de transações EJB, beans stateful, lógica de segurança, ou comportamento que compila mas muda em runtime. Para esses casos, nossa ferramenta detecta, sinaliza e organiza a revisão, mas a decisão é humana. Prometer automação total nesses pontos seria irresponsável em ambiente produtivo.

06 — Plano Faseado & Gates

Da fase mais segura
à mais crítica.

O plano avança em ordem crescente de criticidade. Começamos pelo que é reversível e de baixo impacto, validamos a ferramenta e ganhamos confiança, e só então enfrentamos as transformações que mexem em lógica de negócio. Cada fase termina em um gate de aprovação humana — nenhuma avança sem o "OK" explícito da liderança técnica.

0

Descoberta & Baseline

Criticidade Baixa
Entender a aplicação sem tocá-la. Estabelecer a rede de segurança antes de qualquer mudança.
AçõesInventário de stack e dependências; execução de jdeprscan/jdeps; medição de cobertura de testes; snapshot do comportamento atual (baseline).
Por que primeiroÉ puramente analítica e não-destrutiva. Expõe o tamanho real do trabalho e revela se a cobertura de testes é suficiente para migrar com segurança.
EntregávelRelatório de diagnóstico + proposta de migração faseada.
🔒 Gate 0:  Liderança aprova o plano e confirma que a cobertura de testes é adequada (ou autoriza reforçá-la antes de prosseguir).
1

Infraestrutura & Build

Criticidade Baixa
Preparar o terreno: ambiente, build e pipeline apontando para o JDK alvo, sem alterar código de negócio.
AçõesAtualização de Dockerfile e imagens base; ajuste de GitHub Actions/Jenkins; configuração de toolchain Maven/Gradle; remoção de flags obsoletas (ex.: PermGen, no caminho do JDK 6/8).
Por que cedoTotalmente reversível e isolada do código. Garante que a infraestrutura suporta o JDK 21 antes de migrar a aplicação.
AutomaçãoAlta — recipes de build do OpenRewrite e ajustes de configuração.
🔒 Gate 1:  Build verde no JDK 21 (mesmo que o código ainda use APIs antigas via flags de compatibilidade). Pipeline validado.
2

Modernização de Linguagem

Criticidade Média
Aplicar transformações de linguagem seguras e substituir APIs removidas com mapeamento claro.
AçõesRecipe UpgradeToJava21 do OpenRewrite: substituição de APIs removidas (Thread.stop, etc.), atualização de plugins, ajustes idiomáticos seguros. No caminho do JDK 6, modernizações do 7/8 (try-with-resources, diamond).
Risco controladoSão mudanças com estratégia de migração conhecida e bem testadas pela comunidade. Revisão humana confirma o diff antes do commit.
AutomaçãoAlta — núcleo do OpenRewrite, com dryRun obrigatório antes de aplicar.
🔒 Gate 2:  Revisão do diff completo; build + suíte de testes verdes; comparação com baseline da Fase 0.
3

Namespace Jakarta & Frameworks

Criticidade Alta
A grande virada: javax → jakarta e o salto de versão dos frameworks (Spring Boot 3, Faces 4).
AçõesConversão de namespace via OpenRewrite/Spring Boot Migrator; upgrade de Spring Boot 2→3, PrimeFaces com classifier jakarta; ajuste de propriedades de configuração; recipes WebLogic oficiais quando aplicável.
Por que aquiÉ a mudança mais ampla e a que mais quebra silenciosamente. Só faz sentido depois que linguagem e build já estão estáveis no JDK 21.
AtençãoNem toda biblioteca tem equivalente jakarta; algumas exigem coordenadas Maven diferentes ou Eclipse Transformer. Mapeado caso a caso pelos Stack Profilers.
🔒 Gate 3:  Aplicação sobe no servidor alvo; smoke tests de cada módulo; revisão de segurança (Spring Security 6). Aprovação de arquiteto responsável.
4

Refatoração Semântica Assistida

Criticidade Crítica
Tratar o que a automação não resolve: EJB stateful, transações JTA, beans de UI, lógica que mudou de comportamento.
AçõesA ferramenta NÃO aplica automaticamente. Ela produz um relatório de pontos que exigem decisão humana, sugere abordagens, e organiza a refatoração item a item com validação de paridade funcional.
Por que por últimoÉ onde mora o risco de regressão de negócio. Exige conhecimento do domínio e testes funcionais dedicados. Tratar isso por último concentra o esforço humano onde ele é insubstituível.
EstratégiaIncremental, item a item, com testes de paridade (o comportamento novo bate com o antigo?). Padrão Strangler Fig quando houver redesenho arquitetural.
🔒 Gate 4:  Cada item refatorado validado individualmente; testes funcionais e de paridade aprovados; sign-off de negócio quando aplicável.
5

Validação Final & Cutover

Criticidade Média
Confirmar que a aplicação migrada é funcional e performática, e promover para produção com segurança.
AçõesTestes de regressão completos; testes de performance comparados ao baseline; validação em ambiente de homologação; plano de rollback de produção; cutover monitorado.
EntregávelAplicação em produção no JDK 21 + relatório de auditoria com toda a trilha de decisões e aprovações.
🔒 Gate 5:  Aprovação final da liderança técnica e funcional para cutover de produção, com plano de rollback definido.

Sobre os gates (human-in-the-loop)

Cada gate é um ponto de parada obrigatório. A ferramenta gera um token de aprovação único, vinculado ao estado daquela fase; a fase seguinte só executa mediante esse token, registrado explicitamente pela pessoa responsável. Isso impede que o agente de IA presuma aprovação e cria uma trilha de auditoria — quem aprovou, o quê e quando — essencial para um projeto que toca produção.

07 — Riscos & Mitigações

O que pode dar errado,
e como nos protegemos.

RiscoSeveridadeMitigação
Regressão silenciosa em produção Crítica Mudança compila mas altera comportamento. Mitigação: baseline comportamental na Fase 0, testes de paridade na Fase 4, e exigência de cobertura mínima de testes antes de migrar.
Cobertura de testes insuficiente Crítica Sem testes, não há rede de segurança. Mitigação: o Gate 0 mede a cobertura e pode bloquear a migração até que testes sejam reforçados — decisão consciente da liderança.
Dependência sem versão Jakarta/JDK 21 Alta Biblioteca legada sem equivalente moderno. Mitigação: detecção na Fase 0; uso de Eclipse Transformer para JARs individuais; sinalização explícita quando não houver caminho automático.
EJB / JTA sem tradução automática Crítica Transações container-managed e beans stateful. Mitigação: nunca automatizar; isolar na Fase 4 com refatoração assistida, item a item, e validação de paridade funcional.
Ordem incorreta de transformações Alta Ex.: rodar recipe WebLogic antes do de JDK quebra a limpeza. Mitigação: o Orchestrator impõe a ordem correta de fases e recipes, conforme orientação oficial da Oracle.
Aprovação presumida pelo agente de IA Crítica Agente avança sem aval humano real. Mitigação: token de gate único por fase, registrado explicitamente; nenhuma fase executa sem ele.
Projeto multi-módulo com dependências cruzadas Alta A ordem de migração entre módulos importa. Mitigação: o Orchestrator calcula o grafo de dependências (DAG) e migra em ordem topológica.
Falha de build deixando o repo inconsistente Média Mitigação: cada fase em branch isolada com checkpoint Git; rollback automático em falha de build; nada é aplicado direto na branch principal.

09 — Recomendações Adicionais

Pontos que fazem
a diferença.

  • Tratar cobertura de testes como pré-requisito, não detalhe. A maior proteção contra regressão em produção não é a ferramenta — são os testes. Recomenda-se um diagnóstico de cobertura logo no Gate 0 e, se necessário, uma frente paralela de reforço de testes antes de migrar sistemas críticos.
  • Começar o portfólio pelas aplicações mais simples. Microservices REST stateless primeiro, depois Spring Boot, e por último o conjunto EJB/JSF/WebLogic. Isso constrói confiança organizacional e refina a ferramenta antes dos casos difíceis.
  • Considerar a migração de servidor de aplicação como projeto correlato. Em aplicações WebLogic/EJB, a migração de JDK frequentemente se entrelaça com a do servidor (ex.: WebLogic 14.1.2). Vale alinhar com a liderança se o escopo inclui apenas JDK ou também o servidor.
  • Manter um catálogo vivo de decisões por stack. Cada aplicação migrada gera aprendizado (qual recipe funcionou, qual biblioteca não tinha versão jakarta). Capturar isso em uma base compartilhada acelera as migrações seguintes.
  • Definir claramente "migração" vs. "modernização". A v1 foca em fazer a aplicação rodar no JDK 21. Reescrever EJB para Spring Boot é modernização — um esforço maior que pode ser uma v2. Manter esse limite explícito evita escopo inflado.
  • Planejar para versões futuras. O JDK 25 (LTS, 2025) já é o próximo destino natural. A arquitetura baseada em Knowledge Base e recipes versionados permite estender a ferramenta para novos alvos sem reescrita.
