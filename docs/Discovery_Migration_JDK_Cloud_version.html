1. Contexto e objetivo

O projeto é um servidor MCP (Model Context Protocol) que orquestra a migração de aplicações Java legadas de JDK 6/8 para JDK 21. Ele não reimplementa transformações: orquestra ferramentas maduras (OpenRewrite, Spring Boot Migrator, Eclipse Transformer, jdeprscan/jdeps) e impõe governança human-in-the-loop com gates obrigatórios entre 6 fases (0 a 5).

Hoje o MCP roda localmente via transporte stdio, operando sobre o filesystem da máquina do desenvolvedor. O objetivo é torná-lo 100% remoto, acionado por uma Squad Agêntica em cloud que recebe como parâmetro um repositório do GitHub e conduz a migração de ponta a ponta — criando branches no próprio GitHub, sem nada local.

3. Arquitetura-alvo (cloud)

A arquitetura proposta tem três peças: a Squad Agêntica (orquestrador fino), a Anthropic Console / Messages API como gateway (MCP Connector), e o MCP Server no Render que clona o repositório, transforma e faz push.

┌─────────────────────┐ recebe repoUrl + branch │ SQUAD AGÊNTICA │◄─────────────────────────── usuário/trigger │ (cloud) │ └──────────┬──────────┘ │ POST /v1/messages { mcp_servers:[ url do Render ] } ▼ ┌─────────────────────┐ │ ANTHROPIC API │ MCP Connector = proxy/gateway │ (Console) │ expõe as 12 tools ao Claude automaticamente └──────────┬──────────┘ │ Streamable HTTP (Bearer token) ▼ ┌──────────────────────────────────────────────┐ │ MCP SERVER (Render) │ │ ┌────────────────────────────────────────┐ │ │ │ workspace efêmero: git clone da branch │ │ ◄── árvore real em disco │ │ → roda jdeps/mvn/OpenRewrite aqui │ │ │ └────────────────────────────────────────┘ │ │ estado (config/discovery/plan/report) │ ──► COMMIT+PUSH p/ branch GitHub │ PIN store ............ server-side, NÃO na branch │ └──────────────────────────┬───────────────────┘ │ GitHub API + git push ▼ ┌─────────────────┐ │ GITHUB │ branch de trabalho + PRs + Issues (gates) └─────────────────┘
✔ Por que esse modelo resolve o problema de "filesystem split"
No modelo local remoto ingênuo, o código está numa máquina e o MCP em outra. Aqui não há split: o container clona, transforma e faz push — todo o ciclo é server-side. O desenvolvedor revisa o resultado como Pull Request no GitHub.

4. Estado na branch do GitHub (a sua premissa)

Sua diretriz é: "tudo que se faz em filesystem passe a ser feito na branch de trabalho do GitHub". O objetivo é correto, mas há uma nuance técnica decisiva.

⚠ OpenRewrite e Maven exigem árvore de trabalho real em disco
Não é possível rodar mvn rewrite:run contra "arquivos abstratos numa branch". As transformações operam sobre arquivos físicos. Portanto o caminho de menor risco não é reescrever todo o I/O para a GitHub Contents API — é manter o I/O em disco (no clone) e adicionar um checkpoint de commit+push a cada mutação de estado.

Duas interpretações possíveis

AbordagemComo funcionaRiscoImpacto nos testes
A) Working-tree + push Recomendada Clona a branch para o disco efêmero; readFileSync/writeFileSync continuam iguais; após cada mutação de estado, commit + push para a branch. Em cold-start, re-clona e restaura estado da branch. Baixo Mínimo — fs continua, testes verdes
B) Reescrever I/O para GitHub API Todo readFile/writeFile vira chamada à GitHub Contents/Git Data API contra a branch. Alto Reescrita de ~25 módulos + mocks novos

A recomendação é a (A), introduzindo uma porta de armazenamento (Storage port, padrão hexagonal) com duas implementações: LocalFsStorage (atual, mantém testes) e GitWorkspaceStorage (clone + commit/push). Assim os 56 testes continuam rodando contra a implementação em memória/fs, e o comportamento cloud é um adapter plugável.

// Porta única — todo estado passa por aqui
interface MigrationStorage {
  read(relPath: string): Promise<string | null>
  write(relPath: string, content: string): Promise<void>
  commitState(message: string): Promise<void>   // no LocalFs = no-op
}
// readConfig/writeConfig passam a delegar para storage.read/write
🔴 CRÍTICO — o PIN store NÃO pode ir para a branch

O arquivo .gate-pins.json hoje é deliberadamente isolado do agente. Se ele for commitado na branch de trabalho, a própria Squad — que tem acesso de leitura ao repositório via GitHub API — poderá ler o PIN e auto-aprovar os gates. Isso destrói por completo o modelo human-in-the-loop, que é o coração da governança deste projeto.

Regra inviolável: o PIN store fica server-side (disco efêmero do Render, ou Redis/KV pequeno), nunca na branch, nunca no retorno de nenhuma tool.

⚠ Esclarecido — Squad em board de estados (Jira-like), 100% chat, sem UI de aprovação própria

A Squad opera num board análogo a um board de sprint (Jira), e o agente lê todo o conteúdo do chat/board. Não existe nenhuma UI de aprovação separada do agente dentro da Squad. Isso significa que um PIN exibido apenas no chat seria inútil como segredo.

Porém: confirmado que os usuários são todos funcionários de uma rede corporativa com Microsoft 365 (Teams + e-mail), e a Squad não possui nenhuma integração de leitura com Teams ou e-mail (apenas poderia disparar envio via API/SMTP). Isso restaura exatamente o canal opaco que faltava: o servidor MCP envia o PIN por Teams + e-mail corporativo, sistemas inteiramente fora do contexto do agente; o humano lê o PIN ali e o digita de volta no chat/board para aprovar.

Com esse canal confirmado, a decisão foi simplificar o modelo de gate para PIN único e uniforme nas 6 fases (substituindo a versão híbrida soft/out-of-band avaliada anteriormente). Detalhes na seção 9 e na fase M4 do plano de migração.

Custos da abordagem A

  • Ruído de commits: cada writeConfig vira um commit. Mitigar com mensagens padronizadas (chore(jdk-migration): state checkpoint) e [skip ci], ou agrupando state num único commit por step.
  • Latência: commit+push ≈ 200–500 ms vs. ~1 ms em disco. Aceitável na granularidade de orquestração.
  • Concorrência: 1 migração = 1 branch = 1 sessão → escrita essencialmente serial. Multitenância isola por {owner}/{repo}/{runId}.

5. Render + MCP Connector da Anthropic Console

A Anthropic Console expõe MCP servers remotos via o parâmetro mcp_servers da Messages API (MCP Connector). A Squad faz uma chamada normal à API; a Anthropic atua como proxy e expõe as 12 tools do seu MCP ao Claude automaticamente — a Squad não precisa implementar um cliente MCP.

Ponto essencial: a Console NÃO hospeda o servidor
A Anthropic Console é gateway, não hosting. Você continua precisando hospedar o MCP em uma URL pública (o Render). A Console apenas conecta o Claude da Squad ao seu servidor. 
// chamada típica da Squad
POST https://api.anthropic.com/v1/messages
{
  "model": "claude-opus-4-8",
  "mcp_servers": [{
    "type": "url",
    "url": "https://jdk-migration.onrender.com/mcp",
    "name": "jdk-migration",
    "authorization_token": "Bearer <token>"
  }],
  "messages": [{ "role":"user",
    "content":"Migre o repositório acme/billing-service de JDK 8 para 21" }]
}

Ressalvas do Render

TemaSituaçãoImplicação
RAM no Free tier 512 MB · spin-down após 15 min Bloqueante p/ builds OpenRewrite quer 2 GB+ de heap. Free só valida orquestração (discovery/plano/gate/estado).
Builds reais (execute_phase) Maven + OpenRewrite Exige instância paga (Standard 2 GB+, ~US$ 25/mês) ou offload de build.
Timeout síncrono do connector Tool call é síncrona no turno da API Atenção mvn de 10–15 min estoura limites HTTP → precisa de padrão assíncrono (jobId + polling).
Disco efêmero Workspace some no restart/spin-down OK — estado é restaurado da branch (abordagem A). Workspace é re-clonado.
Cold start Free dorme; acordar + warmup JDK Latência inicial de ~30–60 s na primeira chamada após inatividade.
⚠ Builds longos → tornar execute_phase assíncrono
Em vez de bloquear o turno da API por 15 min, execute_phase deve iniciar um job e retornar { jobId, status:"running" } imediatamente. A Squad faz polling com get_phase_status(jobId) até done/failed. Isso é compatível com o loop agêntico e elimina o risco de timeout.

9. Segurança dos gates: melhora ou piora?

Hoje (local)

PIN em arquivo local, isolado do agente apenas por convenção de "não retornar em tool". Quem roda o Claude Code na máquina tem acesso ao filesystem inteiro.

Cloud (decisão final)

PIN gerado server-side e entregue ao humano via Teams + e-mail corporativo — canais que a Squad não lê. Isolamento por sistema, não por convenção. Mais forte que o modelo local.

✔ Canal corporativo confirmado — Teams + e-mail são opacos ao agente

Confirmado com o responsável do produto: a Squad não possui nenhuma integração de leitura com Teams ou e-mail corporativo (nem bot de canal, nem caixa monitorada, nem Graph API de leitura) — apenas poderia disparar o envio via API/SMTP, nunca ler a resposta. Como todos os usuários da rede corporativa têm Microsoft 365, isso restaura um canal de entrega genuinamente opaco ao agente, mesmo a Squad sendo 100% chat/board no front.

Com esse canal disponível, o modelo híbrido (soft vs. out-of-band) avaliado anteriormente foi descartado em favor de um PIN único e uniforme para as 6 fases — mais simples de operar e auditar, sem abrir mão de garantia técnica em nenhuma fase.

Modelo de gate por PIN via Teams + E-mail (decisão final — 30/06/2026)

EtapaO que aconteceQuem executa
1request_gate_approval(fase) é chamado pelo agente ao concluir a faseSquad (agente)
2Servidor MCP gera PIN de 6 dígitos, grava no CloudSecretStore (server-side, fora da branch) e dispara Teams DM + e-mail ao aprovador responsável (via Graph API / SMTP)Servidor MCP — agente não participa desse envio
3Humano abre Teams ou e-mail (sistemas que a Squad não lê) e vê o PINHumano aprovador
4Humano digita o PIN no chat/board da Squad para confirmar a aprovaçãoHumano (canal lido pelo agente — sem problema aqui, a decisão já foi tomada)
5approve_gate(fase, pin) valida o PIN contra o CloudSecretStore (expiração, HMAC, nome de approver não pode ser bot/claude/ai/agent) e libera a faseServidor MCP

Aplica-se uniformemente às 6 fases (0 a 5) — sem diferenciação de mecanismo por criticidade. A diferenciação de rigor entre fases, se necessária, fica a critério de política operacional (ex.: exigir dois aprovadores nas fases 4/5), não do mecanismo de gate em si.

🔴 Condição inegociável (mantida)

O PIN nunca pode trafegar pelo chat/board antes de o humano o digitar de volta, nunca é commitado na branch, e nunca aparece no retorno de nenhuma tool MCP. O CloudSecretStore vive isolado do GitWorkspaceStorage (que persiste o estado na branch).

Essa garantia depende inteiramente de uma premissa operacional: a Squad nunca deve ganhar, no futuro, nenhuma integração de leitura com o canal de Teams/e-mail usado para envio dos PINs (nem bot de monitoramento geral, nem caixa compartilhada). Qualquer mudança nesse sentido deve revalidar o modelo de gate antes de entrar em produção.

10. Veredito

A arquitetura é viável e recomendada — com 3 condições inegociáveis

A combinação Squad → Anthropic Console (MCP Connector) → MCP no Render → GitHub faz sentido técnico e é o desenho correto para operar múltiplos repositórios de forma agêntica. A maior parte do código (profilers, static analysis, transform engine, reports, ROI) é preservada; a refatoração concentra-se em transporte, porta de armazenamento e operações Git.

  1. PIN server-side, nunca na branch (R1) — condição de existência da governança.
  2. execute_phase assíncrono (R3) — para sobreviver ao timeout síncrono do connector.
  3. Render pago (2 GB+) para builds (R2) — o Free valida orquestração; não roda Maven/OpenRewrite reais.

Estratégia de menor risco para o estado: working-tree + commit/push via uma porta Storage, mantendo os 56 testes verdes. Não reescrever todo o I/O para a GitHub API.

Recomendação de faseamento (resumo): validar primeiro a camada de orquestração no Render Free (transporte HTTP + estado na branch + gates), e só então habilitar os builds pesados em instância paga. O plano detalhado está no documento Migration_Project_cloud_version_proposal.html.
