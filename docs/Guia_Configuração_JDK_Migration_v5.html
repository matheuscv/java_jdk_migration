Ajuste os caminhos para os da sua máquina.

~/.claude.json — copie e ajuste os caminhos
{
  "mcpServers": {
    "jdk-migration": {
      "command": "npx",
      "args": ["-y", "jdk-migration-mcp"],
      "env": {
        // PATH: o MCP precisa achar npx, mvn, java, git e gh (opcional)
        "PATH": "C:\\Program Files\\nodejs;C:\\Users\\SEU_USUARIO\\AppData\\Roaming\\npm;C:\\devtools\\Apache\\maven\\bin;C:\\Program Files\\Zulu\\zulu-21\\bin;C:\\Program Files\\Git\\cmd;C:\\WINDOWS\\system32",
        "PATHEXT": ".COM;.EXE;.BAT;.CMD;.VBS;.WSH;.MSC",
        // JDK 21 (destino) — usado nas Fases 1–5
        "JAVA_HOME": "C:\\Program Files\\Zulu\\zulu-21",
        // JDK de ORIGEM (ex: 8) — usado na Fase 0
        "SOURCE_JAVA_HOME": "C:\\Program Files\\Zulu\\zulu-8",
        "MAVEN_HOME": "C:\\devtools\\Apache\\maven"
      }
    }
  }
}
✔ O que cada campo faz
  • command + args: npx -y jdk-migration-mcp — baixa e executa automaticamente.
  • env.PATH: o campo mais importante. Inclua também o gh se quiser PR automático.
  • env.JAVA_HOME: aponta para o JDK 21 (destino).
  • env.SOURCE_JAVA_HOME: aponta para o JDK de origem (ex: 8).
⚠ Regras do JSON no Windows
  • Use sempre barras duplas \\ nos caminhos.
  • Coloque C:\Program Files\nodejs no início do PATH.

JDK na Fase 0 — em detalhe

Como JAVA_HOME, sourceJdkHome e targetJdkHome são atribuídos

🔵 JAVA_HOME

C:\...\zulu-21

JDK 21 do processo MCP. Nunca é alterado durante a migração.

🟠 sourceJdkHome

C:\...\zulu-8

JDK de origem (6 ou 8). Nunca pode ser 21. Usado na Fase 0 para compilar o legado antes do jdeprscan.

🔵 targetJdkHome

C:\...\zulu-21

JDK destino — sempre 21. Usado nas Fases 1–5 para compilar e testar a aplicação modernizada.

7.1
Cadeia de detecção do sourceJdkHome
Ordem de busca (ex: projeto JDK 8)
1. toolOverrides ............. SOURCE_JAVA_HOME / JAVA_HOME_8
2. variáveis de ambiente ..... SOURCE_JAVA_HOME, JDK_8, JAVA_HOME_1_8
3. paths comuns .............. zulu-8, temurin-8, jdk1.8.0
4. valida a versão ........... precisa ser 8; se for 21 → REJEITA
5. não encontrado ............ PEDE ao usuário via toolOverrides
🟠 Regra inviolável

Um candidato versão 21 nunca é aceito como origem. Configure SOURCE_JAVA_HOME na Seção 2.2 para evitar essa parada.

⚠ Protocolo: branch ANTES do trabalho

Antes de qualquer edição manual, execute:

Terminal — protocolo obrigatório

# 2. Realize o trabalho (mvn, edições, etc.)

Como um gate é aprovado

4 8 2 9 1 3

Aparece no terminal. Informe ao agente junto com "aprovado". Expira em 30 minutos.

🔐 Por que a IA não consegue burlar
  • PIN aleatório (900.000 combinações) em .jdk-migration/.gate-pins.json — nenhuma tool retorna esse arquivo
  • Uso único + expira em 30 min
  • Nomes de sistema bloqueados: bot, claude, ai, agent, ci, gpt…
  • Token assinado por HMAC-SHA256 derivado do caminho do projeto, validade 30 dias

Controle de Branches Git por Fase novo em v5

Cada fase opera em uma branch isolada — rastreabilidade completa e rollback independente por fase

O isolamento por branch é um princípio inegociável do MCP: nenhum commit vai direto na branch principal. Cada fase tem sua própria trilha auditável no Git. O comportamento é diferente para execução automática (via execute_phase) e execução manual (via record_manual_phase).

⚡ Modo automático — execute_phase

  • Branch criada automaticamente antes de qualquer alteração
  • Padrão: jdk-migration/phase-N-YYYYMMDDHHMMSS
  • Transformações aplicadas na branch
  • Commit registrado na branch após build verde
  • PR aberto via gh pr create (se disponível)
  • Rollback: git checkout <baseBranch> automático em falha
  • Branch da fase preservada mesmo no rollback

🔧 Modo manual — record_manual_phase

  • Branch deve ser criada antes de iniciar qualquer trabalho
  • Mesmo padrão: jdk-migration/phase-N-YYYYMMDDHHMMSS
  • Trabalho realizado manualmente na branch
  • Commit manual na branch
  • Registrar no MCP com record_manual_phase passando a branch
  • MCP avança status para awaiting_gate
12.1
Padrão de nomenclatura e inspeção
Exemplo de branches criadas durante a migração
# Lista todas as branches de migração
git branch -a | grep jdk-migration

# Saída esperada:
  jdk-migration/phase-1-20260608090000
  jdk-migration/phase-2-20260608103000
* jdk-migration/phase-3-20260608140000   ← fase atual
  jdk-migration/phase-3-20260608130000   ← tentativa anterior (preservada)

# Inspecionar o histórico de uma fase
git log --oneline jdk-migration/phase-2-20260608103000

# Ver todas as branches incluindo remotas
git log --all --oneline --graph

Branches de fases que falharam também são preservadas — servem como evidência de auditoria e podem ser inspecionadas a qualquer momento.

12.2
syncMigrationBranch — PR automático após cada fase

Após o build e testes passarem no execute_phase, o MCP chama internamente o syncMigrationBranch que tenta abrir um Pull Request da branch da fase para a branch base do projeto.

Comando equivalente executado internamente
gh pr create \
  --head jdk-migration/phase-2-20260608103000 \
  --base main \
  --title "chore(jdk-migration): fase 2 — Modernização de Linguagem" \
  --body "Aplicação do recipe UpgradeToJava21..."
✔ Pré-requisitos para PR automático
  • gh instalado e acessível no env.PATH do ~/.claude.json
  • Autenticado: gh auth status deve retornar ✓
  • Repositório com remote configurado (git remote -v)
12.3
rollback_phase — como reverter uma fase
Usando rollback_phase
# Via Claude Code:
Execute rollback_phase no projeto C:\meu-projeto para a fase 3

# O que acontece internamente:
1. Verifica se fase está em in_progress, awaiting_gate ou failed
2. git checkout <baseBranch>      ← volta para a branch base
3. Status da fase → rolled_back    ← branch da fase preservada
4. Audit report atualizado
⚠ A branch da fase NÃO é deletada

Após o rollback, a branch jdk-migration/phase-3-<ts> permanece no repositório como evidência. Inspecione com git log jdk-migration/phase-3-<ts>.

Fase 5 — Validação Final em Detalhe novo em v5

Checklist de cutover, relatório final e o sign-off duplo obrigatório

A Fase 5 é a última etapa antes do merge para produção. Diferente das fases anteriores, o gate aqui exige sign-off simultâneo da liderança técnica e da liderança funcional. O approve_gate da Fase 5 aciona a geração do audit-report-final.html — um artefato fixo e imutável da migração.

13.1
O gate da Fase 5 requer sign-off duplo
Única fase em que dois aprovadores são esperados — técnico e funcional.
Sequência do gate 5
── Liderança Técnica ─────────────────────────────────────────
1. Revisa o relatório consolidado (todas as fases, riscos, branches)
2. Solicita PIN: request_gate_approval(projectPath, 5)
3. Informa ao Claude: "Aprovado tecnicamente. PIN: 839201. Aprovador: João Silva"
4. approve_gate(projectPath, 5, "João Silva", "839201")

── Liderança Funcional ───────────────────────────────────────
5. Revisa os casos de uso e smoke tests
6. Confirma a aprovação funcional no ticket/email
   → O approve_gate já registrou — não há segunda chamada de ferramenta.
   → O aprovador funcional é documentado no campo note do gate.

── Resultado ─────────────────────────────────────────────────
7. audit-report-final.html gerado em .jdk-migration/
8. Status da Fase 5 → completed
🔒 audit-report-final.html — artefato imutável
  • Gerado apenas uma vez, no approve_gate da Fase 5
  • Contém: trilha completa de todas as fases, gates, aprovadores e timestamps
  • Diferente dos reports incrementais (audit-report-<ts>.html), este não é sobrescrito
  • Deve ser arquivado junto com a aplicação migrada para fins de auditoria
