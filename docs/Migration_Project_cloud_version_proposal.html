1. Princípios da migração

  • Zero regressão funcional: nenhuma fase é concluída com teste vermelho. npm test (56 arquivos) roda no fim de cada fase.
  • Incremental e reversível: cada fase é uma branch isolada; o stdio local continua funcionando até a última fase (paridade dupla).
  • Refatoração por adapter, não por reescrita: introduzir portas (Storage, Git, Transport) e plugar implementações cloud — o núcleo de negócio (profilers, transform-engine, knowledge-base) não muda.
  • Segurança primeiro: o PIN store server-side é tratado na fase de gates, com teste explícito de que o PIN nunca aparece em estado versionado nem em retorno de tool.
  • Build pesado por último: validar orquestração no Render Free antes de habilitar Maven/OpenRewrite em instância paga.

2. Estratégia: Strangler + porta Storage

O coração da migração é isolar os dois pontos de acoplamento ao ambiente local — persistência de estado (src/lib/config.ts) e operações Git (src/orchestrator/git-checkpoint.ts) — atrás de interfaces, com duas implementações cada:

PortaImplementação local (atual)Implementação cloud (nova)
MigrationStorageLocalFsStorage (fs)GitWorkspaceStorage (clone + commit/push)
GitGatewayLocalGitCli (git CLI)GitHubApiGateway (octokit)
TransportStdioServerTransportStreamableHttpTransport
SecretStore (PINs)arquivo local .gate-pins.jsonstore server-side (disco efêmero/KV) — nunca na branch
✔ Por que isso garante baixo risco
Os 56 testes continuam rodando contra as implementações locais/in-memory. A versão cloud é selecionada por configuração de runtime. Em nenhum momento o comportamento local é removido — ele vira o baseline de teste.

Gate M3: PR criado e relatório visível no GitHub; rollback remove a branch de fase sem afetar a base.
M4
Gates via PIN por Teams + E-mail (segurança crítica) Mecanismo único e uniforme para as 6 fases
Risco alto · crítico

Objetivo: garantir que o human-in-the-loop continue tecnicamente real na nuvem. A Squad opera em board de estados (análogo a um board de sprint) e o agente lê todo o chat/board — não há UI de aprovação própria. Confirmado (30/06/2026) que a Squad não tem nenhuma integração de leitura com Teams ou e-mail corporativo (Microsoft 365, presente em toda a rede corporativa) — apenas pode disparar envio via API/SMTP. Isso restaura um canal de entrega genuinamente opaco ao agente. Com esse canal confirmado, a decisão foi por um modelo único de PIN via Teams + e-mail, aplicado uniformemente às 6 fases — substituindo o modelo híbrido (soft/out-of-band) avaliado anteriormente.

Fluxo decidido

  1. request_gate_approval(fase) é chamado pelo agente ao concluir a fase
  2. Servidor MCP gera o PIN, grava no CloudSecretStore (server-side, isolado da branch) e dispara Teams DM + e-mail ao aprovador via Microsoft Graph API — o agente não participa desse envio nem o lê
  3. Humano lê o PIN em Teams/e-mail (sistemas fora do alcance da Squad)
  4. Humano digita o PIN de volta no chat/board da Squad para confirmar a aprovação
  5. approve_gate(fase, pin) valida contra o CloudSecretStore (expiração, HMAC, bloqueio de nomes automatizados) e libera a fase

Tarefas

  • Implementar CloudSecretStore (disco efêmero do Render ou KV/Redis) — isolado do GitWorkspaceStorage, nunca commitado, nunca em retorno de tool
  • Integração de envio via Microsoft Graph API: Teams chat message (1:1) + e-mail (Outlook/Graph sendMail) ao aprovador configurado por fase/projeto
  • Manter validação HMAC do gate token e bloqueio de nomes (bot/claude/ai/agent/...) — já existem, reaproveitados de gate-validator.ts
  • Expiração do PIN (30 min, igual ao modelo local) e invalidação em rollback_phase
  • Aplicar o mesmo mecanismo nas 6 fases (0–5) — sem diferenciação de implementação por criticidade
  • Teste de segurança dedicado: simular leitura de Teams/e-mail pelo agente (mock) e comprovar que nenhuma tool, estado de branch ou log expõe o PIN antes da digitação humana
  • Registrar no CLAUDE.md/runbook a premissa crítica: a Squad nunca deve ganhar integração de leitura sobre o canal de Teams/e-mail usado para os PINs — qualquer mudança futura nesse sentido exige revalidação do modelo de gate
🔴 Condição inegociável
Se um teste demonstrar que a Squad consegue obter o PIN por qualquer caminho — tool, estado da branch, log, ou leitura futura de Teams/e-mail — a fase é reprovada. Toda a garantia de governança do produto depende da Squad permanecer sem acesso de leitura a esses canais.
Gate M4: suíte de segurança comprova isolamento total do PIN; envio via Graph API testado ponta a ponta (Teams + e-mail); aprovação só ocorre com PIN digitado pelo humano após leitura fora do chat.
M5
Execução assíncrona, deploy no Render & Connector Builds longos + integração ponta a ponta com a Squad
Risco médio

Objetivo: habilitar os builds pesados (Maven/OpenRewrite) sem estourar o timeout síncrono do connector, e fechar a integração com a Anthropic Console e a Squad.

Tarefas

  • Tornar execute_phase assíncrono: retorna { jobId, status:"running" }; job roda em background
  • get_phase_status(jobId) reporta progresso até done/failed (a Squad faz polling)
  • Dockerfile multi-stage com JDK 8 + JDK 21 + Maven + Node + git (imagem slim)
  • Deploy no Render: instância paga (≥ 2 GB RAM) para os builds; secrets configurados
  • Registrar o MCP como connector na Anthropic Console (URL + Bearer)
  • Teste end-to-end: Squad chama Messages API → migra repositório sandbox JDK 8→21 completo
  • Validar ROI tracker recebendo tokens via update_phase_costs (sem JSONL local)

Critério de saída

Migração completa de um repositório de teste, ponta a ponta, disparada pela Squad via Anthropic Console, com gates humanos e relatórios no PR.

Gate M5: E2E verde em repo sandbox; nenhum timeout; paridade funcional com a versão local comprovada.

7.5 — M3: Git via GitHub API + PRs (código + config GitHub)

  1. Criar GitHub App (Settings → Developer settings → GitHub Apps) com permissões: Contents: Read & write, Pull requests: Read & write, Issues: Read & write, Metadata: Read-only
  2. Instalar o App no(s) repositório(s) que serão migrados
  3. Gerar private key do App, guardar como secret (não commitar)
  4. Implementar src/adapters/cloud/github-api-gateway.ts usando @octokit/rest + @octokit/auth-app
  5. Mapear: createPhaseBranch → POST /repos/{owner}/{repo}/git/refs; openDraftPr → POST /pulls (draft:true); commentReport → POST /issues/{pr}/comments; rollback → DELETE /git/refs/{ref}
  6. Testar contra repositório sandbox real (criar jdk-migration-sandbox de testes)
Critério para avançar: execução de fase no sandbox cria branch + Draft PR + comentário de relatório visíveis no GitHub real.
