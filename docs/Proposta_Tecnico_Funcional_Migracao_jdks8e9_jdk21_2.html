01 — Contexto & Objetivo

Por que este
projeto é crítico.

Estamos propondo uma ferramenta que altera código de aplicações já em ambiente produtivo. O risco não está em trocar um número de versão — está na heterogeneidade das aplicações-alvo e na possibilidade de regressões silenciosas.

A migração do JDK 6 ou 8 para o JDK 21 atravessa de treze a quinze versões intermediárias. Cada uma acumulou mudanças de linguagem, remoções de API e — sobretudo — três rupturas estruturais: o Sistema de Módulos (JDK 9), a remoção dos módulos Java EE e CORBA (JDK 11) e a encapsulação forte dos internos do JDK (JDK 17).

Somado a isso, frameworks como Spring Boot 3, Jakarta Faces 4 e os próprios servidores de aplicação (WebLogic) evoluíram exigindo o salto do namespace javax.* para jakarta.* — a mudança mais ampla e mais perigosa do ciclo.

Premissa central

Não vamos reinventar a transformação de código. A indústria já consolidou ferramentas maduras: OpenRewrite (com recipes oficiais da Oracle para WebLogic + JDK 21), Spring Boot Migrator e Eclipse Transformer. Nossa solução orquestra esse ecossistema, adiciona diagnóstico especializado por stack e impõe a camada de governança que produção exige.

Stacks cobertas na v1

Microservices REST · Spring Boot (2→3) · Spring Batch · EJB · JSF / PrimeFaces · WebLogic · APIs legadas Java EE

"A ferramenta acelera o que é seguro automatizar e torna visível o que exige julgamento humano — ela nunca confunde as duas coisas."

02 — Arquitetura da Solução

Um MCP orquestrador
com uma Skill de instalação.

A solução combina um servidor MCP — instalável via npm na aplicação-alvo, registrável em qualquer host compatível (VS Code, Claude Code, Cursor) — com uma Skill de instalação que parametriza o comportamento conforme a stack detectada.

Por que MCP, não plugin

Um servidor MCP é um pacote distribuível via npm que roda como processo local. Isso resolve "instalável na aplicação-alvo" sem amarrar a ferramenta a uma IDE específica, e dá acesso nativo ao sistema de arquivos, ao Git e à execução de processos (Maven, Gradle, jdeprscan, OpenRewrite). Um plugin de IDE seria mais limitado e exigiria reimplementação por editor.

A Skill de instalação

No momento da instalação, a Skill conduz a parametrização: detecta a stack, pergunta o que não puder inferir (servidor, versão-alvo de framework, ambiente de CI) e grava um arquivo jdk-migration.config.json que orienta todas as fases seguintes — adaptando a ferramenta genérica ao caso concreto daquela aplicação.

03 — Ferramentas do MCP

Três tools, um fluxo de
confiança progressiva.

A superfície de interação é deliberadamente enxuta. O agente de IA precisa saber quando descobrir, quando planejar e quando executar — sempre mediado por um gate humano.

Tool 01

discover_project

discover_project({ targetJdk }) → { stack, risks, coverageReport }
Escaneia a aplicação sem alterá-la. Identifica build system, stack, versão atual, dependências e todos os pontos de incompatibilidade com o JDK alvo.
  • Detecta Maven / Gradle / Ant e multi-módulo
  • Classifica a stack via Stack Profilers especializados
  • Roda jdeprscan + jdeps contra a versão alvo
  • Mede cobertura de testes (pré-requisito de segurança)
  • Produz inventário de riscos por categoria e stack
Tool 02

build_migration_plan

build_migration_plan({ discoveryId }) → { planId, phases, docPath }
Consolida o diagnóstico em um plano faseado, ordenado por criticidade crescente, com gates definidos. Gera o documento de proposta revisável.
  • Ordena fases da menos à mais crítica
  • Mapeia recipes OpenRewrite aplicáveis por fase
  • Estima esforço e sinaliza revisões obrigatoriamente manuais
  • Define gate e critério de aprovação de cada fase
  • Gera token único por gate para rastreabilidade
Tool 03

execute_phase

execute_phase({ planId, gateToken, phase, dryRun? }) → { branch, pr, log }
Aplica uma única fase aprovada. Exige o token do gate anterior. Cada fase é uma branch isolada com commit e validação de build.
  • Verifica o token de aprovação do gate anterior
  • Suporta dryRun para preview do diff antes de aplicar
  • Roda build + testes após cada fase
  • Rollback automático em falha de build
  • Abre Pull Request com relatório da fase
Auxiliar

ferramentas de apoio

Ferramentas de governança e recuperação usadas pelo Orchestrator e pelo revisor humano.
  • get_phase_status — situação detalhada de cada fase
  • approve_gate — registra a aprovação humana
  • rollback_phase — reverte uma fase aplicada
  • generate_report — relatório consolidado de auditoria

04 — Desafios por Stack

Cada arquitetura tem
seus próprios abismos.

A diversidade das aplicações-alvo é o maior fator de risco. Um microservice stateless é quase trivial; uma aplicação JSF com EJB rodando em WebLogic é um projeto à parte. Os Stack Profilers tratam cada caso com o conhecimento específico que ele exige.

Microservices REST / stateless

Complexidade Baixa

Sem estado, com testes e CI maduros. Migração majoritariamente automatizável via recipe UpgradeToJava21 do OpenRewrite. Caso ideal para iniciar o portfólio e validar a ferramenta antes dos casos complexos.

Spring Boot 2.x

Complexidade Alta

Exige salto para Spring Boot 3 (JDK 17+ obrigatório), com conversão javax → jakarta, mudanças no Spring Security 6 e no RestTemplate. Spring Boot Migrator automatiza o mecânico; configuração de segurança exige revisão obrigatória.

Spring Batch

Complexidade Alta

Spring Boot 3 traz o Spring Batch 5 com novo modelo de configuração de jobs. @EnableBatchProcessing legado e beans como JobRepository seguem novos padrões. Lógica de chunk e restart precisa de validação funcional dedicada.

EJB

Complexidade Crítica

Beans stateful, transações container-managed (JTA), chamadas remotas RMI/IIOP e JNDI não têm tradução automática segura. Exige redesenho assistido e validação de paridade funcional rigorosa, item a item.

JSF / PrimeFaces

Complexidade Crítica

Migração de namespace nas XHTML, managed beans de javax.faces.bean para CDI, PrimeFaces com classifier jakarta, e regras de client-ID que mudaram no Faces 4. Partes de UI exigem trabalho manual sem recipe disponível.

WebLogic

Complexidade Alta

A Oracle fornece recipes OpenRewrite oficiais (rewrite-weblogic) para WebLogic 14.1.2/15.1.1 + JDK 21. Regra obrigatória: recipe de JDK primeiro, depois recipe de WebLogic. Conectores JCA e realms de segurança proprietários requerem atenção manual.

Limite honesto de automação

OpenRewrite e Spring Boot Migrator resolvem com excelência o trabalho mecânico: troca de namespace, bump de versões, substituição de APIs com mapeamento claro. Eles não resolvem mudanças semânticas — redesenho de transações EJB, beans stateful, lógica de segurança, ou comportamento que compila mas muda em runtime. Para esses casos, nossa ferramenta detecta, classifica e organiza a revisão, mas a decisão é humana. Prometer automação total nesses pontos seria irresponsável em produção.

05 — Plano Faseado & Gates

Da fase mais segura
à mais crítica.

O plano avança em ordem crescente de criticidade. Começamos pelo que é reversível e de baixo impacto — validamos a ferramenta e ganhamos confiança — e só então enfrentamos as transformações que mexem em lógica de negócio. Cada fase termina em um gate de aprovação humana.

0

Descoberta & Baseline

Criticidade Baixa

Entender a aplicação sem tocá-la. Estabelecer a rede de segurança antes de qualquer mudança.

AçõesInventário de stack e dependências; execução de jdeprscan/jdeps; medição de cobertura de testes; snapshot comportamental (baseline).
Por que primeiroPuramente analítica e não-destrutiva. Expõe o tamanho real do trabalho e revela se a cobertura de testes é suficiente para migrar com segurança.
EntregávelRelatório de diagnóstico + proposta de migração faseada para aprovação da liderança.
🔒 Gate 0:  Liderança aprova o plano e confirma cobertura de testes adequada (ou autoriza reforçá-la antes de prosseguir).
1

Infraestrutura & Build

Criticidade Baixa

Preparar o terreno: ambiente, build e pipeline apontando para o JDK 21, sem alterar código de negócio.

AçõesAtualização de Dockerfile e imagens base; ajuste de GitHub Actions / Jenkins; configuração de toolchain Maven/Gradle; remoção de flags obsoletas (ex.: PermGen no caminho JDK 6/8).
Por que cedoTotalmente reversível e isolada do código. Garante que a infra suporta o JDK 21 antes de migrar a aplicação.
AutomaçãoAlta — recipes de build do OpenRewrite e ajustes de configuração.
🔒 Gate 1:  Build verde no JDK 21 com flags de compatibilidade. Pipeline de CI validado no novo JDK.
2

Modernização de Linguagem

Criticidade Média

Aplicar transformações de linguagem seguras e substituir APIs removidas com mapeamento claro.

AçõesRecipe UpgradeToJava21 do OpenRewrite: substituição de APIs removidas (Thread.stop, etc.), atualização de plugins, ajustes idiomáticos seguros. No caminho do JDK 6, modernizações do 7/8 (try-with-resources, diamond operator).
RiscoMudanças com estratégia de migração conhecida, bem testadas pela comunidade. dryRun obrigatório para revisão do diff antes de aplicar.
AutomaçãoAlta — núcleo do OpenRewrite. Revisão humana do diff completo obrigatória antes do commit.
🔒 Gate 2:  Revisão do diff completo; build + suíte de testes verdes; comparação com baseline da Fase 0.
3

Namespace Jakarta & Frameworks

Criticidade Alta

A grande virada: javax → jakarta e o salto de versão dos frameworks.

AçõesConversão de namespace via OpenRewrite / Spring Boot Migrator; upgrade Spring Boot 2→3; PrimeFaces com classifier jakarta; recipes WebLogic oficiais quando aplicável; ajuste de propriedades de configuração.
Por que aquiÉ a mudança mais ampla e mais perigosa. Só faz sentido após linguagem e build já estáveis no JDK 21. Nem toda biblioteca tem equivalente jakarta — casos mapeados pelos Stack Profilers.
AtençãoAlgumas libs exigem coordenadas Maven diferentes ou Eclipse Transformer. WebLogic: recipe de JDK obrigatoriamente primeiro.
🔒 Gate 3:  Aplicação sobe no servidor alvo; smoke tests por módulo; revisão de Spring Security 6. Aprovação de arquiteto responsável.
4

Refatoração Semântica Assistida

Criticidade Crítica

Tratar o que automação não resolve: EJB stateful, transações JTA, lógica que muda de comportamento.

AçõesA ferramenta NÃO aplica automaticamente. Produz relatório de pontos que exigem decisão humana, sugere abordagens, organiza a refatoração item a item com validação de paridade funcional.
Por que por últimoÉ onde mora o risco de regressão de negócio. Exige conhecimento do domínio e testes funcionais dedicados. Concentrar aqui o esforço humano onde ele é insubstituível.
EstratégiaIncremental, item a item, com testes de paridade. Padrão Strangler Fig quando houver redesenho arquitetural (ex.: EJB → Spring).
🔒 Gate 4:  Cada item refatorado validado individualmente; testes funcionais e de paridade aprovados; sign-off de negócio quando aplicável.
5

Validação Final & Cutover

Criticidade Média

Confirmar que a aplicação migrada é funcional e performática, e promover para produção com segurança.

AçõesTestes de regressão completos; performance comparada ao baseline; validação em homologação; plano de rollback de produção; cutover monitorado.
EntregávelAplicação em produção no JDK 21 + relatório de auditoria com trilha completa de decisões e aprovações.
🔒 Gate 5:  Aprovação final da liderança técnica e funcional para cutover, com plano de rollback definido e testado.

Sobre os gates (human-in-the-loop)

Cada gate é um ponto de parada obrigatório. A ferramenta gera um token de aprovação único, vinculado ao estado daquela fase; a fase seguinte só executa mediante esse token, registrado explicitamente pela pessoa responsável. Isso impede que o agente de IA presuma aprovação e cria uma trilha de auditoria rastreável — quem aprovou, o quê e quando — essencial para um projeto que toca produção.

06 — Riscos & Mitigações

O que pode dar errado,
e como nos protegemos.

RiscoSeveridadeMitigação
Regressão silenciosa em produçãoCríticaMudança compila mas altera comportamento. Baseline comportamental na Fase 0, testes de paridade na Fase 4 e cobertura mínima de testes como pré-requisito inegociável.
Cobertura de testes insuficienteCríticaSem testes, não há rede de segurança. Gate 0 mede a cobertura e pode bloquear a migração até que testes sejam reforçados — decisão consciente da liderança.
Dependência sem versão Jakarta / JDK 21AltaDetecção na Fase 0; uso de Eclipse Transformer para JARs individuais; sinalização explícita quando não houver caminho automático.
EJB / JTA sem tradução automática seguraCríticaNunca automatizar. Isolado na Fase 4 com refatoração assistida, item a item, e validação de paridade funcional obrigatória.
Ordem incorreta de transformaçõesAltaEx.: recipe WebLogic antes do de JDK quebra a limpeza. O Orchestrator impõe a ordem correta conforme orientação oficial da Oracle.
Aprovação presumida pelo agente de IACríticaToken de gate único por fase, registrado explicitamente. Nenhuma fase executa sem o token correto — não é possível inferir ou fabricar.
Multi-módulo com dependências cruzadasAltaOrchestrator calcula o grafo de dependências (DAG) e migra módulos em ordem topológica.
Build deixando repo em estado inconsistenteMédiaCada fase em branch isolada com checkpoint Git. Rollback automático em falha de build. Nada é aplicado direto na branch principal.

08 — Recomendações Adicionais

Pontos que fazem
a diferença.

  • Tratar cobertura de testes como pré-requisito. A maior proteção contra regressão não é a ferramenta — são os testes. Se necessário, uma frente paralela de reforço de testes antes de migrar sistemas críticos.
  • Começar o portfólio pelos mais simples. Microservices REST stateless primeiro, depois Spring Boot, por último EJB/JSF/WebLogic. Isso constrói confiança organizacional e refina a ferramenta antes dos casos difíceis.
  • Alinhar migração de JDK com migração de servidor. Em aplicações WebLogic/EJB, a migração de JDK frequentemente se entrelaça com a do servidor. Vale definir com a liderança se o escopo inclui apenas o JDK ou também o servidor de aplicação.
  • Manter um catálogo vivo de decisões por stack. Cada aplicação migrada gera aprendizado (qual recipe funcionou, qual biblioteca não tinha versão jakarta). Capturar em base compartilhada acelera as migrações seguintes.
  • Separar migração de modernização. A v1 faz a aplicação rodar no JDK 21. Reescrever EJB para Spring Boot é modernização — escopo de uma possível v2. Manter esse limite explícito evita escopo inflado.
  • Planejar para versões futuras. O JDK 25 (LTS, 2025) já é o próximo destino natural. A arquitetura baseada em Knowledge Base e recipes versionados permite estender a ferramenta sem reescrita.
