1 Antes de começar

Pré-requisitos

Garanta que os itens abaixo estão instalados e disponíveis no PATH da máquina onde o servidor MCP vai rodar.

☕ JDK 21

Necessário para o jdeprscan e jdeps rodarem análise estática sobre bytecode e código-fonte. Sem JDK 21, o jdeprscanItemCount será 0.

java -version # Esperado: openjdk 21.x.x jdeprscan --help # Deve existir no PATH

🐙 Git ≥ 2.30

O orquestrador cria branches isoladas por fase e executa rollback automático via Git em caso de falha no build. O projeto-alvo deve ser um repositório Git.

git --version # Esperado: git version 2.30+ git init # no projeto-alvo, se ainda não for repo

🟩 Node.js ≥ 20 LTS

Runtime do servidor MCP. Requerido para compilar e executar o dist/mcp-server/index.js.

node --version # Esperado: v20.x.x ou v22.x.x npm --version # Esperado: 10.x.x+

🔨 Maven ou Gradle

O build system do projeto-alvo deve estar disponível. O detector de stack suporta pom.xml (Maven), build.gradle (Gradle) e build.xml (Ant — detecção parcial).

mvn --version # Maven gradle --version # Gradle
ℹ️
OpenRewrite, Spring Boot Migrator e Eclipse Transformer são baixados automaticamente pelo execute_phase quando necessário. Não precisam ser instalados manualmente.
2 Servidor MCP

Instalação e Build

Clone o repositório da ferramenta, instale as dependências e compile o TypeScript. O servidor expõe as tools via protocolo MCP (JSON-RPC sobre stdio).

1

Clone e instale dependências

Faça o clone do repositório java_jdk_migration e instale os pacotes npm.

terminal
# Clone do repositório da ferramenta MCP git clone https://github.com/seu-org/java_jdk_migration.git cd java_jdk_migration # Instala todas as dependências npm install
2

Compile o TypeScript

Gera os arquivos JavaScript em dist/. O servidor MCP executa o artefato compilado — não o TypeScript diretamente.

terminal
# Verifica tipos sem emitir arquivos (opcional, mas recomendado) npm run typecheck # Compila TypeScript → dist/ npm run build # Confirma que o servidor MCP responde ao handshake echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.0.1"}}}' | node dist/mcp-server/index.js
✅
Resposta esperada: {"result":{"protocolVersion":"2024-11-05","capabilities":{"tools":{"listChanged":true}},"serverInfo":{"name":"jdk-migration","version":"0.1.0"}}...}
3

Registre o servidor no seu cliente MCP

Configure seu agente de IA (Claude, Cursor, VS Code Copilot, etc.) para usar o servidor. O transporte é stdio — o cliente lança o processo e se comunica via stdin/stdout.

~/.claude/settings.json (exemplo para Claude Code)
{ "mcpServers": { "jdk-migration": { "command": "node", "args": ["/caminho/para/java_jdk_migration/dist/mcp-server/index.js"], "type": "stdio" } } }
⚠️
Use o caminho absoluto para o index.js. Caminhos relativos falham dependendo do diretório de trabalho do cliente.
4

Rode os testes para confirmar integridade

Antes de usar em produção, valide que os 31 testes unitários passam.

terminal
npm run test:unit # Esperado: # ✓ tests/unit/errors.test.ts (6 tests) # ✓ tests/unit/skill-detect.test.ts (11 tests) # ✓ tests/unit/config.test.ts (8 tests) # ✓ tests/unit/process-runner.test.ts (6 tests) # Tests: 31 passed
3 Projeto-alvo

Configuração do Projeto-Alvo

A Skill de instalação detecta automaticamente o build system e a stack tecnológica do projeto e grava o arquivo jdk-migration.config.json na raiz. Você pode revisar e editar este arquivo antes de avançar.

Como inicializar a configuração

💡
A inicialização é feita via código TypeScript/JS, invocando install() da Skill. No fluxo normal com um agente de IA, basta pedir ao agente para executar a discovery — ele chama discover_project que inicializa automaticamente se o config não existir.
inicialização manual via Node.js
// _init.ts — rode com: npx tsx _init.ts import { install } from './src/skill/install.ts' const result = await install('/caminho/absoluto/do/seu/projeto-java') console.log('Stack detectada:', result.config.stack) console.log('JDK fonte:', result.config.sourceJdk) console.log('Precisa de ajuste manual:', result.requiresHumanInput) // → grava /seu/projeto/jdk-migration.config.json

Estrutura do jdk-migration.config.json

jdk-migration.config.json
{ "sourceJdk": "8", // "6" ou "8" "targetJdk": "21", // sempre "21" (fixo) "stack": ["spring-boot", "spring-batch"], "buildSystem": "maven", // "maven" | "gradle" | "ant" "appServer": null, // "weblogic" | "jboss" | "tomcat" | "liberty" | null "multiModule": false, "modulePaths": [], // ex: ["api/", "service/", "web/"] "ciSystem": null, // "github-actions" | "jenkins" | "gitlab-ci" | null "testCoverageThreshold": 80, // % mínimo de cobertura antes de executar fases "dryRunBeforeExecute": true, // força dryRun antes de fases alta/crítica "phases": { "0": { "status": "pending", "gateToken": null, "approvedBy": null }, "1": { "status": "pending", "gateToken": null, "approvedBy": null }, // ... fases 2 a 5 com o mesmo esquema } }

Referência dos campos

sourceJdk
"6" | "8"
Versão de origem do JDK. Detectada automaticamente via maven.compiler.source, java.version ou sourceCompatibility. JDK 7 é normalizado para "8" (recipes cobrem ambos).
targetJdk
"21" (fixo)
Versão de destino. Sempre "21". Não altere — os recipes de OpenRewrite são parametrizados para JDK 21.
stack
StackType[]
Array de tecnologias detectadas. Valores válidos: "rest", "spring-boot", "spring-batch", "ejb", "jsf", "weblogic", "jakarta". Pode ser editado manualmente se a detecção automática estiver incompleta.
buildSystem
"maven" | "gradle" | "ant"
Sistema de build detectado pela presença de pom.xml, build.gradle ou build.xml. Ant tem suporte parcial — detecção apenas, sem transformações automáticas.
appServer
"weblogic" | "jboss" | "tomcat" | "liberty" | null
Servidor de aplicação alvo. Quando "weblogic", recipes Oracle oficiais (com.oracle.weblogic.rewrite) são adicionados à Fase 3. Atenção: recipe JDK deve rodar ANTES do recipe WebLogic.
multiModule
boolean
Indica se o projeto é multi-módulo Maven/Gradle. Quando true, informe os caminhos em modulePaths. O orquestrador aplicará as transformações em cada módulo.
testCoverageThreshold
number (0–100)
Cobertura mínima de testes exigida antes de executar fases de média/alta criticidade. Padrão: 80. Abaixo deste valor, o gate não será emitido automaticamente.
dryRunBeforeExecute
boolean
Quando true (padrão), o sistema força um dryRun antes de aplicar transformações em stacks de alta/crítica complexidade. Nunca desative em EJB ou JSF.
4 Orquestração

Fluxo Completo de Migração

O fluxo segue uma máquina de estados estrita. Nenhuma fase avança sem o token de gate da fase anterior. Cada fase é executada em uma branch Git isolada.

Tool

Gate 0

Fase 1–5

→
Gate N

🚨
Princípio inegociável: nunca avançar de fase sem token de gate aprovado. O token é único por fase e registrado explicitamente via approve_gate. Qualquer tentativa de pular o gate resulta em erro GATE_TOKEN_INVALID.

Passo a passo detalhado

1

🔍 Instalar e detectar a stack

Inicializa o jdk-migration.config.json no projeto-alvo. Detecta automaticamente JDK, build system e tecnologias. Se requiresHumanInput: true, edite o config manualmente antes de prosseguir.

// Via agente de IA — simplesmente peça: "Execute a descoberta do projeto em /meu/projeto/java" // O agente chamará internamente: discover_project({ projectPath: "/meu/projeto/java" })
2

📊 Executar a Descoberta

Escaneia o projeto sem alterar nenhum arquivo. Roda análise estática (jdeprscan, jdeps), correlaciona com a knowledge base de APIs removidas e gera o discovery-report.json.

// Saída esperada (resumida): { "detectedStacks": ["spring-boot", "spring-batch"], "sourceJdk": "8", "riskSummary": { "critical": 1, "high": 2, "estimatedEffortDays": 10 }, "knowledgeCorrelation": [ { "severity": "critical", "apiPattern": "javax.xml.bind" }, { "severity": "high", "apiPattern": "sun.misc.BASE64Encoder" } ] }
3

📋 Construir o Plano de Migração

Consolida o discovery report em um plano faseado. Seleciona os recipes corretos para cada fase baseado na stack detectada. Gera o migration-plan.json. Leia atentamente antes de aprovar.

build_migration_plan({ projectPath: "/meu/projeto/java" }) // O plano lista para cada fase: // - recipes que serão aplicados (OpenRewrite, SBM, Eclipse Transformer) // - estimativa de dias // - nível de automação // - itens que requerem revisão manual // - descrição do gate (o que o humano precisa validar)
4

🔐 Aprovar o Gate da Fase 0

Revisado o plano e confirmado que a cobertura de testes é adequada, um humano aprova o Gate 0. O sistema emite um token criptográfico que habilita a execução da Fase 1. O nome do aprovador não pode ser bot, automation, ci, cd, system ou auto.

approve_gate({ projectPath: "/meu/projeto/java", phaseNumber: 0, approvedBy: "TechLead", // nome real — não pode ser "bot" ou "ci" notes: "Plano revisado. Cobertura 85%. Ok para avançar." }) // Retorna: { "gateToken": "26ed4203c721...", "validFor": "phase_1" }
5

🧪 dryRun antes de cada fase crítica

Para fases de alta ou crítica complexidade, execute primeiro com dryRun: true. O sistema mostra o diff exato sem modificar nenhum arquivo. Obrigatório para EJB, JSF e qualquer fase 3+.

execute_phase({ projectPath: "/meu/projeto/java", phaseNumber: 1, gateToken: "26ed4203c721...", dryRun: true // preview sem aplicar }) // Retorna diffSummary e recipesWouldApply // Sem modificação nos arquivos do projeto
6

⚙️ Executar a fase

Com o dryRun revisado e aprovado, execute a fase de fato. O orquestrador: (1) verifica que o workdir Git está limpo, (2) cria uma branch isolada jdk-migration/phase-N, (3) aplica os recipes, (4) compila o projeto, (5) roda os testes. Em caso de falha, o rollback é automático.

execute_phase({ projectPath: "/meu/projeto/java", phaseNumber: 1, gateToken: "26ed4203c721...", dryRun: false // aplica de verdade }) // Branch criada: jdk-migration/phase-1 // Status da fase atualizado: in_progress → completed
7

🔁 Repetir para fases 2–5

Repita o ciclo approve_gate → dryRun → execute_phase para cada fase subsequente. Monitore o progresso a qualquer momento com get_phase_status. Em caso de problema, use rollback_phase para reverter via Git.

// Verificar status de todas as fases: get_phase_status({ projectPath: "/meu/projeto/java" }) // Reverter uma fase em caso de problema: rollback_phase({ projectPath: "/meu/projeto/java", phaseNumber: 2 }) // Gerar relatório final com trilha de auditoria: generate_report({ projectPath: "/meu/projeto/java" })
5 Referência

MCP Tools — Referência Completa

Todas as tools são expostas via protocolo MCP e chamadas pelo agente de IA. Cada tool aceita JSON e retorna JSON.

🔍
discover_project
Escaneia o projeto sem alterar nenhum arquivo. Primeira tool a ser executada.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java a ser analisado
Retorna: stack detectada, JDK, build system, análise estática (jdeprscan/jdeps), correlação com knowledge base de APIs removidas, risk summary, pré-requisitos. Salva .jdk-migration/discovery-report.json.
📋
build_migration_plan
Consolida o discovery em um plano faseado com recipes selecionados por stack.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java
Pré-condição: discover_project deve ter sido executado antes (lê o discovery-report.json).

Retorna: array de PhasePlan com recipes, estimativa de dias, automationLevel, riskItems, manualReviewItems e gateDescription para cada fase.
⚙️
execute_phase
Aplica uma fase aprovada. Cria branch Git isolada. Suporta dryRun.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java
phaseNumber number (0–5) ✓ Número da fase a executar
gateToken string — (Fase 0) Token emitido pelo approve_gate da fase anterior. Obrigatório para fases 1–5.
dryRun boolean false Se true, exibe o diff sem modificar arquivos. Use sempre antes de fases críticas.
⚠️
O working directory Git deve estar limpo (git status sem modificações). Caso contrário, o erro GIT_DIRTY_WORKDIR é lançado.
📊
get_phase_status
Retorna o status atual de todas as fases da migração.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java
Status possíveis por fase: pending · in_progress · awaiting_gate · approved · completed · failed · rolled_back
🔐
approve_gate
Registra aprovação humana e emite o token para a fase seguinte.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java
phaseNumber number (0–5) ✓ Número da fase sendo aprovada
approvedBy string ✓ Nome do aprovador humano. Proibido: bot, automation, ci, cd, system, auto.
notes string opcional Observações do aprovador para a trilha de auditoria
↩️
rollback_phase
Reverte uma fase via Git. Retorna ao commit baseline da fase.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java
phaseNumber number (0–5) ✓ Fase a ser revertida. Deve estar em estado failed ou completed.
🚨
Rollback manual via Git também é sempre possível: git checkout jdk-migration/phase-N-base.
📄
generate_report
Gera relatório consolidado com trilha de auditoria completa da migração.
ParâmetroTipoReqDescrição
projectPath string ✓ Caminho absoluto da raiz do projeto Java
Inclui: todas as fases executadas, tokens de gate, aprovadores, timestamps, branches Git criadas, recipes aplicados, resultado dos builds e testes. Salvo em .jdk-migration/audit-report.json.
6 Arquitetura

Fases de Migração

A migração é dividida em 6 fases com criticidade crescente. Cada fase tem um gate obrigatório com critérios claros de aceitação.

0

Descoberta & Baseline

● Baixa

Escaneia o projeto sem alterá-lo. Classifica a stack, roda jdeprscan/jdeps, correlaciona com a knowledge base. Gera o inventário de riscos.

Automação: Alta — apenas leitura
Recipes: nenhum (análise apenas)
Gate 0: Humano aprova o plano e confirma cobertura de testes ≥ threshold.
1

Infraestrutura & Build

● Baixa

Atualiza o build system (versão do plugin Maven/Gradle), configura o target JDK 21 no compilador. Sem alteração de código-fonte.

Automação: Alta
Runners: build-updater
Gate 1: Build verde no JDK 21 (mvn compile / gradle build).
2

Modernização de Linguagem

● Média

Aplica recipes OpenRewrite para JDK 21: APIs depreciadas, Thread#stop, sun.misc.*, finalize(), atualiza imports e padrões de linguagem.

Recipe principal: org.openrewrite.java.migrate.UpgradeToJava21
Gate 2: Diff revisado pelo tech lead + todos os testes verdes.
3

Namespace Jakarta & Frameworks

● Alta

Converte javax.* → jakarta.*, migra Spring Boot 2→3 (via SBM), Spring Batch 4→5, e WebLogic (recipes Oracle oficiais).

Runners: openrewrite + sbm
Ordem: JDK recipe → WebLogic recipe (obrigatório)
Gate 3: Aplicação sobe no servidor alvo; smoke tests passam.
4

Refatoração Semântica Assistida

● Crítica

EJB stateful/stateless, JTA, CDI, JSF/PrimeFaces. Sem automação total — a ferramenta sinaliza, organiza e orienta; a refatoração é feita pelo desenvolvedor.

Automação: Baixa (apenas sinaliza)
dryRun: Obrigatório
Gate 4: Paridade funcional item a item validada pelo time.
5

Validação Final & Cutover

● Média

Suite de testes de regressão completa, testes de performance, validação de segurança. Geração do relatório final de auditoria. Preparação para produção.

Recipes: nenhum (validação)
Relatório: generate_report
Gate 5: Sign-off da liderança técnica e funcional.
7 Compatibilidade

Stacks Suportadas

Cada stack tem um profiler especializado que analisa os riscos específicos e seleciona os recipes adequados.

Stack Identificador Complexidade Automação Engines utilizadas
Microservices REST stateless "rest" Baixa Alta OpenRewrite (UpgradeToJava21)
Spring Boot 2.x "spring-boot" Alta Alta OpenRewrite + Spring Boot Migrator
Spring Batch "spring-batch" Alta Alta OpenRewrite (SpringBatch4To5Migration)
EJB (stateful/stateless/MDB) "ejb" Crítica Baixa Apenas sinaliza — refatoração manual
JSF / PrimeFaces "jsf" Crítica Parcial OpenRewrite (namespace) + revisão manual de UI
WebLogic "weblogic" Alta Alta OpenRewrite (oracle/rewrite-recipes)
Jakarta EE (namespace) "jakarta" Média Alta OpenRewrite + Eclipse Transformer
🚨
EJB e JSF são stacks críticas. Nunca aplique transformações semânticas automaticamente em EJB/JTA. A ferramenta apenas sinaliza os pontos de atenção e organiza a revisão humana. O dryRun é mandatório nestas stacks.
8 Troubleshooting

Códigos de Erro

Todos os erros são instâncias de MigrationError com um code identificador. Use o código para diagnosticar e corrigir sem precisar ler o stack trace.

🔍
STACK_NOT_DETECTED
Nenhum build system detectado no caminho informado. Sem pom.xml, build.gradle ou build.xml.
✅ Verifique se projectPath aponta para a raiz do projeto Java (onde fica o arquivo de build).
📄
CONFIG_NOT_FOUND
O jdk-migration.config.json não existe, ou o discovery-report.json não foi gerado ainda.
✅ Execute discover_project primeiro. O config é criado automaticamente na primeira discovery.
🔐
GATE_TOKEN_INVALID
O token de gate informado é inválido, expirado ou não corresponde à fase anterior.
✅ Execute approve_gate(phaseN-1) para gerar um novo token válido.
🚫
PHASE_OUT_OF_ORDER
Tentativa de executar uma fase sem que a fase anterior esteja approved. A máquina de estados é estrita.
✅ Verifique o status com get_phase_status e aprove as fases na ordem correta (0 → 1 → 2 ...).
🔨
BUILD_FAILED
O build do projeto falhou após a aplicação dos recipes. O rollback automático já foi executado.
✅ Leia o failureOutput no retorno do erro. Corrija o problema apontado e tente novamente. Considere executar dryRun para entender o diff antes.
🗂️
GIT_DIRTY_WORKDIR
O working directory Git possui alterações não commitadas ou em staging. O orquestrador recusa executar sobre workdir sujo.
✅ Faça git commit ou git stash para limpar o workdir antes de executar a fase.
🔒
LOCK_FILE_EXISTS
O arquivo .jdk-migration/lock existe, indicando que outra execução está em andamento (ou travou).
✅ Verifique se há outro processo rodando. Se for um lock órfão, remova manualmente: rm .jdk-migration/lock.
9 Checklist

Validação da Configuração

Marque cada item abaixo para confirmar que a configuração está completa e correta antes de avançar para a primeira fase de migração.

🔧 Ambiente

  • JDK 21 instalado e no PATH Verificar com java -version e jdeprscan --help
  • Node.js ≥ 20 LTS instalado Verificar com node --version
  • Git ≥ 2.30 instalado Verificar com git --version
  • Maven ou Gradle disponível O build system do projeto-alvo deve ser executável no PATH

📦 Servidor MCP

  • Dependências instaladas (npm install) Pasta node_modules/ deve existir na raiz do repositório
  • Build compilado sem erros (npm run build) Pasta dist/ deve existir com mcp-server/index.js
  • Testes unitários passando (npm run test:unit) 31 testes devem passar: errors, skill-detect, config, process-runner
  • Handshake MCP respondendo corretamente Enviar initialize via stdin e receber protocolVersion: 2024-11-05
  • Servidor registrado no cliente MCP (Claude/Cursor/VS Code) Caminho absoluto para dist/mcp-server/index.js no settings.json

🎯 Projeto-alvo

  • Projeto-alvo é um repositório Git git status funciona na raiz do projeto-alvo
  • Working directory limpo antes da primeira fase git status não deve mostrar alterações pendentes
  • jdk-migration.config.json gerado na raiz Executar discover_project cria o config automaticamente
  • Stack detectada corretamente no config Verificar o campo stack — editar manualmente se necessário
  • sourceJdk correto no config Deve ser "6" ou "8" conforme o projeto-alvo

🔍 Discovery e Plano

  • discover_project executado com sucesso Arquivo .jdk-migration/discovery-report.json deve existir
  • Risk summary revisado e compreendido Itens critical e high devem ter plano de ação definido
  • build_migration_plan executado e plano revisado Cada fase do plano revisada: recipes, estimativa de dias, itens manuais
  • Cobertura de testes ≥ threshold definido no config Padrão: 80%. Stacks EJB/JSF recomendam ≥ 90% antes de avançar
  • Gate 0 aprovado por humano identificado (não "bot") approve_gate({ phaseNumber: 0, approvedBy: "NomeReal" }) executado
🎉
Configuração completa! Todos os itens foram validados. O projeto está pronto para iniciar a Fase 1 de migração. Lembre-se: execute sempre dryRun: true antes de cada fase de alta criticidade.
