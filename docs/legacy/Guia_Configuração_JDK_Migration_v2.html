Pré-requisitos

O que deve estar instalado antes de começar

1.1
Node.js ≥ 18 e npm ≥ 9
O MCP server roda em Node.js. Versões anteriores não suportam ESM nativo.
💻 Terminal local 👤 Humano
PowerShell / Terminal
node --version    # deve ser v18 ou superior
npm --version     # deve ser 9 ou superior
✔ Validação
  • Saída de node --version começa com v18, v20 ou v24
  • Saída de npm --version é 9.x ou superior
1.2
Claude Code CLI instalado e autenticado
O MCP é consumido pelo Claude Code. Certifique-se de que o CLI está instalado e logado.
💻 Terminal local 👤 Humano
Terminal
claude --version   # verifica instalação
claude mcp list   # lista MCPs ativos (pode estar vazio)

Se claude não for reconhecido: instale em claude.ai/code ou via extensão VS Code / JetBrains.

1.3
Git inicializado no projeto-alvo
O MCP cria branches isoladas por fase. O projeto Java precisa ser um repositório Git.
💻 Raiz do projeto-alvo 👤 Humano
Terminal — raiz do projeto Java
git status   # confirma que é um repositório git
# Se não for: git init && git add . && git commit -m "baseline"
✔ Validação
  • git status não retorna "not a git repository"
  • Existe ao menos um commit na branch principal
1.4
JDK 21 disponível na máquina
O MCP detecta o JDK via JAVA_HOME para rodar análise estática (jdeprscan, jdeps).
💻 Terminal local 👤 Humano
Terminal
java --version
# Exemplo de saída esperada: openjdk 21.x.x ...
⚠ Se JAVA_HOME não estiver configurado
  • Windows: $env:JAVA_HOME = "C:\Program Files\Java\jdk-21"
  • macOS/Linux: export JAVA_HOME=$(/usr/libexec/java_home -v 21)
2

Instalação do MCP via npm

Duas abordagens — escolha conforme o seu caso de uso

2.1
Abordagem A — npx (recomendada · sem instalação manual)
O npx baixa automaticamente o pacote na primeira execução, cacheia e executa. Não requer npm install manual. Sempre usa a versão latest do registry.
📄 Arquivo de configuração 👤 Humano (edita o arquivo)

Nenhum comando de instalação é necessário — o npx -y no arquivo de configuração é suficiente. Veja a Seção 3 para o arquivo settings.json.

✔ Vantagens
  • Zero manutenção — sempre usa a versão mais recente automaticamente
  • Sem risco de conflito de dependências no projeto-alvo
  • Ideal para uso pontual em um único projeto de migração
2.2
Abordagem B — Instalação global (offline ou versão fixada)
Instala o pacote globalmente para uso sem internet após o download.
💻 Terminal local 👤 Humano
Terminal
npm install -g jdk-migration-mcp

# Verificar instalação:
jdk-migration-mcp --version   # opcional — confirma que o binário está no PATH

# Atualizar quando sair nova versão:
npm update -g jdk-migration-mcp
⚠ Quando usar instalação global

Use se precisar de versão fixada, ambiente sem internet ou múltiplos projetos simultâneos.

3

Configuração no Claude Code (projeto-alvo)

Registrar o MCP para que o agente possa usar as tools de migração

3.1
Opção A — Escopo de projeto (recomendado)
Cria o arquivo .claude/settings.json na raiz do projeto Java que será migrado. O MCP fica ativo apenas neste projeto e pode ser commitado no repositório para o time.
📁 Raiz do projeto-alvo 👤 Humano

Crie a pasta e o arquivo:

PowerShell — raiz do projeto Java
New-Item -ItemType Directory -Force .claude
.claude/settings.json — Abordagem npx
{
  "mcpServers": {
    "jdk-migration": {
      "command": "npx",
      "args": ["-y", "jdk-migration-mcp"]
    }
  }
}

Se usou instalação global (Abordagem B da Seção 2):

.claude/settings.json — Abordagem global
{
  "mcpServers": {
    "jdk-migration": {
      "command": "jdk-migration-mcp"
    }
  }
}
✔ Estrutura esperada
  • Arquivo em SEU_PROJETO/.claude/settings.json
  • JSON válido (sem vírgulas extras ou aspas simples)
3.2
Opção B — Escopo global (disponível em qualquer projeto)
Registra o MCP globalmente via CLI. Disponível em todos os projetos da máquina.
💻 Terminal (qualquer diretório) 👤 Humano
Terminal
claude mcp add jdk-migration -- npx -y jdk-migration-mcp

Este comando edita ~/.claude/settings.json automaticamente. Equivalente a editar o arquivo manualmente com o JSON da Opção A.

4

Validação da Conexão MCP

Confirmar que o agente enxerga as tools antes de começar a migração

4.1
Reiniciar o Claude Code e verificar o MCP
O Claude Code carrega as configurações de MCP na inicialização da sessão. Após criar o settings.json, é obrigatório reiniciar.
💻 Terminal — raiz do projeto-alvo 👤 Humano
Terminal — dentro do projeto-alvo
# 1. Encerrar a sessão atual do Claude Code (se aberta)
# 2. Iniciar nova sessão NA RAIZ DO PROJETO-ALVO:
claude

# 3. Dentro do Claude Code, verificar MCPs ativos:
! claude mcp list
✔ Saída esperada
  • jdk-migration aparece na lista com status connected
  • 7 tools disponíveis: discover_project, build_migration_plan, execute_phase, get_phase_status, approve_gate, rollback_phase, generate_report
⚠ MCP não aparece?
  • Confirme que o Claude Code foi aberto na raiz do projeto (não em subpasta)
  • Valide o JSON com: Get-Content .claude/settings.json | ConvertFrom-Json (PowerShell)
  • Verifique se o npx consegue baixar o pacote: npx -y jdk-migration-mcp --version

Configuração da Skill no Projeto-alvo

Inicializar o arquivo de configuração jdk-migration.config.json antes de começar

5.1
Solicitar ao Claude Code que rode a Skill de instalação
A Skill detecta automaticamente o build system, stack e versão de JDK do projeto. Ela grava jdk-migration.config.json na raiz e adiciona entradas ao .gitignore.
💬 Chat do Claude Code 🤖 Claude Code
Mensagem para o Claude Code
Configure o JDK Migration MCP neste projeto.
O projeto está em: /caminho/absoluto/do/projeto
JDK de origem: 8
Build system: maven

Arquivos criados automaticamente pela Skill:

jdk-migration.config.json — exemplo gerado
{
  "sourceJdk": "8",
  "targetJdk": "21",
  "stack": ["spring-boot"],
  "buildSystem": "maven",
  "appServer": null,
  "multiModule": false,
  "modulePaths": [],
  "ciSystem": null,
  "testCoverageThreshold": 80,
  "dryRunBeforeExecute": true,
  "phases": { /* status de cada fase — gerenciado automaticamente */ }
}
✔ Validação
  • Arquivo jdk-migration.config.json criado na raiz do projeto
  • Pasta .jdk-migration/ criada (metadados e relatórios)
  • .gitignore contém .jdk-migration/ e jdk-migration.config.json
5.2
Revisar e ajustar o config manualmente (se necessário)
Se o projeto tiver múltiplos módulos, app server (WebLogic/JBoss) ou stack mista, edite o config antes de prosseguir.
📄 Editor de texto 👤 Humano
Campos opcionais para ajuste manual
"stack": ["spring-boot", "ejb"],   // rest | spring-boot | spring-batch | ejb | jsf | weblogic
"appServer": "weblogic",             // null | weblogic | jboss | tomcat
"multiModule": true,
"modulePaths": ["core", "api", "web"],
"ciSystem": "github-actions",        // github-actions | jenkins | gitlab-ci | null
"testCoverageThreshold": 70           // % mínimo de cobertura para Gate 0

Referência Completa das 7 Tools MCP

Parâmetros, quando usar e como validar cada tool

discover_project
Escaneia a aplicação Java sem alterar nenhum arquivo. Detecta build system, stack, JDK atual, dependências e incompatibilidades com JDK 21 via jdeprscan, jdeps e análise de código-fonte. Deve ser a primeira tool executada — nenhuma fase avança sem um relatório de descoberta. Salva o resultado em .jdk-migration/discovery-report.json.
💬 Claude Code 🤖 Claude Code (executa a tool)
Leitura only
ParâmetroTipoDescrição
projectPath string obrigatório Caminho absoluto da raiz do projeto Java a ser analisado
Como solicitar ao Claude Code
Execute a tool discover_project no projeto em /c/meu-projeto-java
Exemplo de retorno (resumido)
{
  "detectedStacks": ["spring-boot"],
  "buildSystem": "maven",
  "sourceJdk": "8",
  "riskSummary": {
    "critical": 2, "high": 5, "medium": 12, "low": 8,
    "manualReviewRequired": true,
    "estimatedEffortDays": 18
  },
  "prerequisites": {
    "jdk21Available": true,
    "gitAvailable": true
  },
  "savedReportPath": ".jdk-migration/discovery-report.json"
}
✔ Validação
  • Nenhum arquivo do projeto foi modificado
  • Arquivo .jdk-migration/discovery-report.json criado com o relatório
  • Campo prerequisites.jdk21Available é true
  • Stack detectada corresponde ao projeto (revise se incorreta antes de prosseguir)
build_migration_plan
Consolida o relatório de diagnóstico em um plano faseado revisável, classificado por criticidade, com gates de aprovação definidos e estimativa de esforço por fase. O documento gerado deve ser revisado e aprovado pelo humano antes de qualquer execução.
💬 Claude Code 🤖 Claude Code (executa a tool)
Planejamento
ParâmetroTipoDescrição
projectPath string obrigatório Caminho absoluto da raiz do projeto Java
Como solicitar ao Claude Code
Gere o plano de migração para o projeto em /c/meu-projeto-java
✔ Validação
  • Retorna plano com 6 fases numeradas (0–5)
  • Cada fase tem criticidade, esforço estimado e descrição do gate
  • Campo totalEstimatedDays faz sentido para o porte do projeto
  • Revise manualmente os itens de manualItems em cada fase
execute_phase
Aplica uma fase de migração aprovada. Cria uma branch Git isolada por fase. Em caso de falha de build, o rollback é automático. Suporta dryRun para preview do diff sem aplicar mudanças — obrigatório em fases de alta/crítica criticidade. A Fase 0 não requer token. As fases 1–5 exigem o gateToken emitido por approve_gate.
💬 Claude Code 🤖 Claude Code (executa a tool)
Modifica código
ParâmetroTipoDescrição
projectPath string obrigatório Caminho absoluto da raiz do projeto Java
phaseNumber number 0–5 obrigatório Número da fase a executar
gateToken string opcional Token emitido por approve_gate ao aprovar a fase anterior. Fase 0 não requer.
dryRun boolean padrão: false Se true, exibe diff sem modificar arquivos. Recomendado antes de fases críticas.
Execução típica — dry-run primeiro, depois execução real
# 1. Preview sem modificar (obrigatório para fases críticas)
Execute a fase 1 em dryRun=true no projeto /c/meu-projeto com gateToken: abc123...

# 2. Após revisar o diff, executar de fato
Execute a fase 1 no projeto /c/meu-projeto com gateToken: abc123...
✔ Validação após execução
  • Status retornado é awaiting_gate (não failed)
  • Campo branchName mostra a branch criada (ex: jdk-migration/phase-1-a3f8...)
  • Build passou: sem erros de compilação na branch da fase
  • Diff gerado é coerente com o escopo da fase
⚠ Rollback automático

Se o build falhar após a transformação, o MCP reverte automaticamente ao commit base e define status failed. Nenhuma ação manual necessária.

get_phase_status
Retorna o status atual de todas as 6 fases, incluindo tokens de gate, datas de aprovação, branches Git associadas e quem aprovou cada fase. Use para monitorar o progresso da migração a qualquer momento.
💬 Claude Code 🤖 Claude Code (executa a tool)
Leitura only
ParâmetroTipoDescrição
projectPath string obrigatório Caminho absoluto da raiz do projeto Java
Como solicitar ao Claude Code
Qual o status de todas as fases da migração em /c/meu-projeto?
Status possíveis por fase
"pending"        → Ainda não iniciada
"in_progress"    → Transform em execução
"awaiting_gate"  → Aguarda aprovação humana
"approved"       → Gate aprovado, pronto para próxima fase
"failed"         → Build falhou — rollback automático executado
"rolled_back"    → Revertida manualmente
approve_gate
Registra a aprovação humana de uma fase e emite o token que libera a fase seguinte. Nunca deve ser chamado por automação ou CI/CD — o campo approverName é validado e nomes como "bot", "ci", "automation" são rejeitados. O token emitido é único e vinculado ao projeto e fase.
💬 Claude Code 👤🤖 Humano solicita · Claude executa
Gate obrigatório
ParâmetroTipoDescrição
projectPath string obrigatório Caminho absoluto da raiz do projeto Java
phaseNumber number 0–5 obrigatório Número da fase a aprovar
approverName string obrigatório Nome completo do responsável. Mínimo 2 caracteres. Não pode ser "bot", "ci", "auto", etc.
Como solicitar ao Claude Code
Aprove o gate da fase 0 no projeto /c/meu-projeto.
Aprovador: João Silva

  "approvedBy": "João Silva",
  "gateToken": "eyJwaGFzZSI6MCwidHMi...",  ← use em execute_phase da Fase 1
  "message": "Gate da Fase 0 aprovado. Use o gateToken para liberar a Fase 1..."
}
✔ Validação
  • Status retornado é approved
  • gateToken não está vazio — copie-o para usar em execute_phase
  • get_phase_status mostra a fase como approved
rollback_phase
Reverte uma fase aplicada via Git, restaurando o projeto ao estado anterior. Funciona para fases com status in_progress, awaiting_gate ou failed. A branch da fase é preservada como histórico — apenas o checkout retorna ao base. Não requer token de gate.

Reversível
ParâmetroTipoDescrição

phaseNumber number 0–5 obrigatório Número da fase a reverter
Como solicitar ao Claude Code
Reverta a fase 2 no projeto /c/meu-projeto
✔ Validação
  • Status retornado é rolled_back
  • restoredBranch indica a branch base para onde retornou
  • git branch mostra a branch da fase ainda existindo (preservada como histórico)
generate_report
Gera o relatório consolidado HTML de auditoria da migração, com trilha completa de decisões, aprovações por fase, arquivos modificados e itens manuais em aberto. Salvo em .jdk-migration/audit-report-{timestamp}.html.

Troubleshooting

Erros comuns e como resolver

!
MCP não aparece em claude mcp list
Diagnóstico
# Verificar JSON do settings.json (PowerShell)
Get-Content .claude/settings.json | ConvertFrom-Json

# Testar npx manualmente
npx -y jdk-migration-mcp

# Verificar localização do settings.json
# Deve estar em: RAIZ_DO_PROJETO/.claude/settings.json
# NÃO em: subpasta / pasta pai / home directory
!
Erro GATE_TOKEN_INVALID

O token de gate é válido por sessão e vinculado ao projeto + fase. Causas comuns: (1) token copiado incorretamente, (2) token de fase errada, (3) token expirado (sessão encerrada). Solução: execute approve_gate novamente para emitir novo token.

!
Erro STACK_NOT_DETECTED

O MCP não encontrou pom.xml, build.gradle ou build.xml na raiz do projeto. Verifique se o projectPath aponta para a raiz correta (onde fica o arquivo de build). Para projetos multi-módulo, use a raiz do módulo pai.

!
Fase vai para status failed (build quebrou)

O rollback é automático — o projeto volta ao estado anterior sem ação manual. Para investigar: verifique o log de erro retornado na tool, corrija o problema identificado (pode ser dependência incompatível ou recipe inapropriado para o projeto), e reexecute a fase com dryRun=true primeiro para validar antes de aplicar.

Verificar estado após falha
Qual o status das fases em /c/meu-projeto?
# Fase com status "failed" → projeto já está no estado base (rollback automático)
# A branch da fase existe para inspeção do diff: git log --oneline jdk-migration/phase-N-*
!
Erro EPERM no Windows ao remover diretório temporário

Processo Git pode estar com handle aberto no diretório. Solução: feche o IDE/Git GUI e remova manualmente o diretório temporário listado no erro (padrão: %TEMP%\jdkm-*).

PowerShell — remover lock órfão
Remove-Item -Recurse -Force "$env:TEMP\jdkm-*"
