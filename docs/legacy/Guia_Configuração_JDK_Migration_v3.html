JDK Migration MCP
Guia Completo v3

Passo a passo para instalar, configurar e operar o MCP de migração Java JDK 6/8 → 21. Inclui as novas features de detecção de ambiente, scanner de Dockerfile/CI, aprovação com PIN e registro manual de fase.

v0.3.11 · npmjs.com JDK 6/8 → 21 Claude Code · MCP 🔐 PIN Gate Human-in-the-loop

🆕 Novidades da v3 em relação à v2

  • Detecção automática do ambiente de build — discover_project agora detecta Java, Maven, Git e Gradle na máquina e exibe tabela no report HTML. Se alguma ferramenta obrigatória não for encontrada, pede o caminho ao usuário antes de continuar. novo
  • Scanner de Dockerfile, docker-compose e CI pipelines — incompatibilidades de imagem base (ex: openjdk:8, jre-java-8:cielo) são detectadas automaticamente e exibidas em seção dedicada do report. novo
  • Gate com PIN de confirmação obrigatório — fluxo de aprovação agora exige PIN de 6 dígitos gerado por request_gate_approval e informado pelo humano. Claude fisicamente não consegue aprovar um gate de forma autônoma. novo
  • Tool record_manual_phase — escape hatch para registrar fases executadas fora do MCP (ex: após falha ambiental), com suporte a steps detalhados em uma única chamada. novo
  • Tool request_gate_approval — nova tool obrigatória antes de approve_gate. Gera e exibe o PIN. novo
  • Retry automático de fase failed — execute_phase reseta automaticamente o status sem necessidade de editar o config manualmente. melhorado
  • Resolução de .cmd no Windows — spawn ENOENT e spawn EINVAL para Maven/Git/Gradle corrigidos na fonte. corrigido
  • reportMode padrão: phase-gate-step — relatório HTML gerado a cada step concluído, mantendo "Progresso dos Steps" sempre atualizado. alterado
  • 11 tools MCP no total (v2 tinha 7). expandido

Pré-requisitos

O que deve estar instalado antes de começar

1.1
Node.js ≥ 18 e npm ≥ 9
O MCP server roda em Node.js. Versões anteriores não suportam ESM nativo.
💻 Terminal local👤 Humano
PowerShell / Terminal
node --version    # deve ser v18 ou superior
npm --version     # deve ser 9 ou superior
✔ Validação
  • Saída de node --version começa com v18, v20 ou superior
  • Saída de npm --version é 9.x ou superior
1.2
Claude Code CLI instalado e autenticado
O MCP é consumido pelo Claude Code. Certifique-se de que o CLI está instalado e logado.
💻 Terminal local👤 Humano
Terminal
claude --version
claude mcp list   # lista MCPs ativos (pode estar vazio na primeira vez)

Se claude não for reconhecido: instale em claude.ai/code ou via extensão VS Code / JetBrains.

1.3
Git inicializado no projeto-alvo
O MCP cria branches Git isoladas por fase. O projeto Java precisa ser um repositório Git com ao menos um commit.
💻 Raiz do projeto-alvo👤 Humano
Terminal — raiz do projeto Java
git status
# Se não for repositório: git init && git add . && git commit -m "baseline"
1.4
JDK 21 e Maven disponíveis na máquina
O MCP detecta as ferramentas automaticamente durante o discover_project. Se não encontrar, pedirá os caminhos. Mas é melhor ter tudo no PATH antes de começar.
💻 Terminal local👤 Humano
Terminal
java --version    # deve mostrar openjdk 21.x
mvn --version     # Apache Maven 3.x
git --version     # git version 2.x

Na v3, o MCP detecta automaticamente onde estão instalados consultando variáveis de ambiente (JAVA_HOME, MAVEN_HOME, M2_HOME) e o PATH. Se algo não for encontrado, a ferramenta pergunta ao humano antes de continuar. Veja a Seção 4 para configuração de PATH no Windows.

Instalação do MCP via npm

Duas abordagens — escolha conforme o seu caso de uso

2.1
Abordagem A — npx (recomendada · zero instalação)
O npx -y no arquivo de configuração faz o download e execução automaticamente. Não requer npm install manual. Sempre usa a versão latest do registry.
📄 Arquivo de configuração👤 Humano
✔ Vantagens
  • Zero manutenção — sempre usa a versão mais recente automaticamente
  • Sem risco de conflito de dependências no projeto-alvo
2.2
Abordagem B — Instalação global (versão fixada)
Instala globalmente para uso offline ou quando se quer fixar uma versão específica.
💻 Terminal local👤 Humano
Terminal
npm install -g jdk-migration-mcp

# Atualizar para versão mais recente:
npm update -g jdk-migration-mcp

4

Configuração de PATH e Ambiente no Windows

Essencial para o MCP encontrar Maven, Java e Git automaticamente

4.1
Configurar PATH do MCP server em ~/.claude.json
O processo do MCP server no Windows não herda o PATH do usuário. É necessário declará-lo explicitamente no env do servidor MCP. Sem isso ocorrerá spawn ENOENT ou spawn EINVAL ao tentar executar mvn, java ou git.
📄 ~/.claude.json👤 Humano (edita o arquivo)
~/.claude.json — trecho do servidor jdk-migration
{
  "mcpServers": {
    "jdk-migration": {
      "env": {
        "PATH": "C:\\Program Files\\nodejs;C:\\Users\\SEU_USUARIO\\AppData\\Roaming\\npm;C:\\devtools\\Apache\\maven\\bin;C:\\Program Files\\Zulu\\zulu-21\\bin;C:\\Program Files\\Git\\cmd;C:\\WINDOWS\\system32",
        "PATHEXT": ".COM;.EXE;.BAT;.CMD;.VBS;.WSH;.MSC",
        "JAVA_HOME": "C:\\Program Files\\Zulu\\zulu-21",
        "MAVEN_HOME": "C:\\devtools\\Apache\\maven"
      }
    }
  }
}
⚠ Importante
  • Node.js é obrigatório no PATH — sem C:\Program Files\nodejs, o comando npx não é encontrado e o MCP server não inicia. Este é o erro mais comum de configuração.
  • Inclua também C:\Users\SEU_USUARIO\AppData\Roaming\npm para pacotes npm globais
  • Ajuste todos os caminhos para os diretórios reais de instalação na sua máquina
  • Após editar ~/.claude.json, reinicie o Claude Code para que as variáveis sejam carregadas
  • Use barras duplas \\ nos caminhos Windows dentro do JSON

A partir da v0.3.6, o MCP resolve mvn.cmd, gradle.bat e similares automaticamente no PATH configurado — não é mais necessário apontar o executável completo.

discover_project
Escaneia a aplicação Java sem alterar nenhum arquivo. Na v3 inclui: detecção de ferramentas do ambiente (Java, Maven, Git), análise estática de código e scanner de Dockerfile/docker-compose/CI pipelines. Deve ser a primeira tool executada.

Retorna o status atual de todas as 6 fases, branches Git, datas e responsáveis por aprovações.

Fluxo de Aprovação de Gate com PIN

Como funciona a proteção contra aprovação autônoma pelo agente

9.1
Por que o PIN é necessário?
Antes da v3, a proteção existia apenas por convenção (regras no CLAUDE.md). Claude poderia chamar approve_gate com um nome qualquer e o MCP executaria. O PIN cria uma barreira técnica que o agente não consegue transpor sozinho.
🔐 O que garante a proteção
  • PIN gerado com randomInt(100000, 999999) — 900.000 possibilidades
  • Salvo em .jdk-migration/.gate-pins.json — não retornado por nenhuma tool MCP
  • Claude não tem acesso de leitura a esse arquivo via tools do MCP
  • Expira em 30 minutos e é consumido após uso (uso único)
  • Sem o PIN correto, approve_gate lança erro GATE_TOKEN_INVALID
9.2
Fluxo passo a passo do gate com PIN
Sequência completa de aprovação
── CLAUDE (automático após execute_phase) ──────────────────
1. Claude chama request_gate_approval(projectPath, phaseN)
   → MCP gera PIN aleatório, salva em disco, exibe:

   ══════════════════════════════════════════════════
     PIN DE APROVAÇÃO — FASE 2
     ➜  4 8 2 9 1 3
   ══════════════════════════════════════════════════
   Expira em 30 minutos.

── VOCÊ (humano) ───────────────────────────────────────────
2. Revisa o resultado da fase (diff, build, testes)
3. Lê o PIN na tela
4. Responde com aprovação explícita:
   "Aprovado. PIN: 482913. Aprovador: Matheus Castro"

── CLAUDE (após receber aprovação + PIN) ───────────────────
5. Chama approve_gate(projectPath, 2, "Matheus Castro", "482913")
   → MCP valida o PIN contra o arquivo em disco
   → PIN consumido (uso único)
   → gateToken emitido para a Fase 3
⚠ PIN expirado ou incorreto?

Peça ao Claude para chamar request_gate_approval novamente. Um novo PIN de 6 dígitos será gerado. O anterior é invalidado automaticamente.

Troubleshooting

Problemas comuns e como resolver

T0
MCP server não inicia — npx não encontrado
O Claude Code não consegue iniciar o servidor MCP porque npx não está no PATH configurado.
⚠ Causa

O env do MCP em ~/.claude.json não contém C:\Program Files\nodejs. Sem Node.js no PATH, npx jdk-migration-mcp falha antes mesmo de carregar qualquer tool.

✔ Solução
  • Adicione C:\Program Files\nodejs no início do PATH no env do MCP (veja Seção 4)
  • Adicione também C:\Users\SEU_USUARIO\AppData\Roaming\npm para pacotes globais
  • Reinicie o Claude Code após editar ~/.claude.json
  • Teste: claude mcp list deve mostrar jdk-migration: ✓ Connected
T1
spawn ENOENT — mvn / java / git não encontrado
O processo do MCP server não enxerga o PATH do sistema operacional.
⚠ Causa

O env do MCP server em ~/.claude.json não tem o diretório bin do Maven/Java no PATH.

✔ Solução
  • Adicione o caminho bin do Maven e JDK à variável PATH no env do MCP (veja Seção 4)
  • Adicione "PATHEXT": ".COM;.EXE;.BAT;.CMD;..." no mesmo env
  • Reinicie o Claude Code após editar ~/.claude.json
T2
spawn EINVAL — .cmd não executa no Windows
Arquivo .cmd encontrado mas não pode ser executado diretamente sem cmd.exe.

Corrigido automaticamente na v0.3.6+. Se ainda ocorrer, verifique se está usando a versão mais recente: npm update -g jdk-migration-mcp

T3
GATE_TOKEN_INVALID — PIN incorreto ou expirado
✔ Solução
  • PIN expirou (30 min)? Peça ao Claude para chamar request_gate_approval novamente
  • PIN digitado incorretamente? Verifique o PIN exibido na tela e informe os 6 dígitos exatos
  • approverName bloqueado? Não use "bot", "claude", "assistant", "ai" — use seu nome real
T4
Fase presa em failed após falha

Na v3, execute_phase reseta automaticamente o status failed → pending antes de tentar novamente. Nenhuma edição manual do config é necessária. Se o trabalho foi feito manualmente, use record_manual_phase para registrar.

T5
Ferramenta não encontrada pelo discover_project
✔ Solução
  • O MCP informa qual ferramenta está faltando e pede o caminho
  • Informe via toolOverrides: { "MAVEN_HOME": "C:\\caminho\\para\\maven" }
  • Ou configure o PATH do MCP server em ~/.claude.json (Seção 4) — solução definitiva
T6
Dockerfile/CI não aparece no relatório

O scanner busca arquivos até 5 níveis de profundidade a partir da raiz do projeto. Certifique-se de que o projectPath aponta para a raiz real do repositório (onde ficam Dockerfile e .gitlab-ci.yml). A seção "Containers & CI" só aparece se houver arquivos de infraestrutura encontrados.
