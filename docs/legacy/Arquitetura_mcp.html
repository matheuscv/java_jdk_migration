As 6 Fases da Migração

Cada fase tem criticidade, nível de automação e gate de aprovação definidos

0
Descoberta & Baseline
Criticidade: Baixa
Sem alteração de código Automação Alta
  • Inventário completo de riscos via jdeprscan + jdeps + source scanner
  • Correlação com knowledge base de APIs removidas
  • Gate: humano aprova plano e confirma cobertura de testes ≥ 80%
1
Infraestrutura & Build
Criticidade: Baixa
Build Updater Automação Alta
  • Atualiza compiler target: 1.8 → 21 no pom.xml / build.gradle
  • Ajusta plugin versions e toolchain do Maven/Gradle
  • Gate: build verde no JDK 21 sem erros de compilação
2
Modernização de Linguagem
Criticidade: Média
OpenRewrite Automação Alta
  • Recipe UpgradeToJava21 — APIs deprecadas/removidas
  • Modernização de sintaxe: try-with-resources, diamond operator, etc.
  • Gate: diff revisado pelo responsável técnico + testes verdes
3
Namespace Jakarta & Frameworks
Criticidade: Alta
OpenRewrite SBM Automação Alta
  • javax.* → jakarta.* em todos os namespaces
  • Spring Boot 2→3, Spring Security 6, Spring Batch 5
  • Gate: aplicação sobe no servidor alvo + smoke tests passando
4
Refatoração Semântica Assistida
Criticidade: Crítica
Manual EJB / JSF Automação: Baixa/Nenhuma
  • EJB @Stateful: apenas sinaliza — nenhuma automação, refatoração manual
  • JSF / PrimeFaces: namespace automático, revisão visual de componentes
  • Gate: paridade funcional confirmada item a item pelo time técnico e funcional
5
Validação Final & Cutover
Criticidade: Média
generate_report Automação Alta
  • Relatório consolidado com trilha de auditoria completa
  • Validação de performance e segurança no ambiente de homologação
  • Gate: sign-off formal da liderança técnica E funcional

MCP Tools — Referência Rápida

As 7 tools expostas pelo servidor ao agente de IA

discover_project
Escaneia a aplicação sem alterá-la. Classifica stack, roda jdeprscan/jdeps/source-scanner e correlaciona com a knowledge base. Deve ser o primeiro comando executado.
Input: projectPath Sem alteração de código
build_migration_plan
Consolida o relatório de descoberta em plano faseado com gates. Inicializa o state machine e persiste jdk-migration.config.json. Requer discover_project anterior.
Input: projectPath Gera migration-plan.json
execute_phase
Aplica uma fase aprovada. Valida gate token, cria branch Git isolada, aplica transformações, executa build + testes, commita. Suporta dryRun. Gera audit report automaticamente.
Exige gate token (fase > 0) Cria branch isolada Rollback automático em falha
approve_gate
Registra aprovação humana e emite token para a fase seguinte. Nunca pode ser chamado por automação — rejeita approverNames que identifiquem sistemas automatizados. Gera audit report.
Human-only Emite token assinado
get_phase_status
Retorna o status atual de todas as fases: pending, in_progress, awaiting_gate, approved, completed, failed ou rolled_back. Lê o state do jdk-migration.config.json.
Read-only Sem efeito colateral
rollback_phase
Reverte uma fase aplicada via Git — volta para a branch base. A branch de fase é preservada como registro de auditoria. Disponível enquanto fase estiver em awaiting_gate ou approved.
Checkout para branch base Branch fase preservada
generate_report
Gera relatório HTML consolidado com trilha de auditoria, registro de riscos, status de fases, itens manuais e plano de execução. Também gerado automaticamente em cada fase e gate.
Input: projectPath audit-report-<ts>.html

Stacks Suportadas

Nível de automação e estratégia por tecnologia

Stack Complexidade Automação Ferramentas Observações
Microservices REST stateless Baixa ✔ Alta OpenRewrite UpgradeToJava21 Caso mais simples — totalmente automatizável
Spring Boot 2.x Alta ✔ Alta (mecânica) OpenRewrite + SBM + jakarta recipes Revisão obrigatória em Spring Security 6
Spring Batch Alta ✔ Alta OpenRewrite + validação funcional de jobs Novo modelo de jobs no Batch 5 requer validação
EJB (Stateful / Stateless) Crítica ✘ Baixa Apenas sinaliza · Nenhuma recipe automática @Stateful: redesenho obrigatório manual. JTA complexo.
JSF / PrimeFaces Crítica ◑ Parcial Namespace automático · UI manual @ManagedBean removido · revisão visual de XHTML obrigatória
WebLogic Alta ✔ Alta oracle/rewrite-recipes (oficial Oracle) Recipe JDK deve rodar ANTES do recipe WebLogic

Princípios Inegociáveis

Regras de governança que o MCP impõe programaticamente

🔐
Gate Token Obrigatório
Nenhuma fase avança sem o token emitido pelo approve_gate da fase anterior. Token único por fase, registrado com timestamp e assinatura do aprovador.
🛑
EJB: Sem Automação Semântica
Transformações semânticas em EJB/JTA nunca são aplicadas automaticamente — o MCP apenas sinaliza e organiza a revisão humana obrigatória.
🌿
Branch Git Isolada por Fase
Cada fase opera em uma branch jdk-migration/phase-N-<ts>. Nunca há commit direto na branch principal. Rollback automático em falha de build.
🧪
dryRun Antes de Stacks Críticas
O parâmetro dryRun: true é obrigatório antes de executar fases em stacks de alta ou crítica complexidade — preview do diff sem aplicar mudanças.
📋
Ordem de Recipes Importa
Recipe JDK (UpgradeToJava21) sempre executa antes do recipe WebLogic — esta é a ordem oficial Oracle e não pode ser invertida.
👤
approve_gate é Humano-Only
A tool rejeita automaticamente approverNames que identifiquem sistemas (ex: "claude", "bot", "automation"). O aprovador deve ser uma pessoa real, identificada pelo nome.

Artefatos Gerados no Projeto-Alvo

Estrutura de diretórios criada pelo MCP na raiz da aplicação em migração

projeto-alvo/
├── jdk-migration.config.json ← state machine + configuração da migração
└── .jdk-migration/
├── discovery-report.json ← resultado do discover_project
├── migration-plan.json ← plano faseado do build_migration_plan
├── audit-report-2026-06-03T10-00-00.html ← snapshot após Fase 0
├── audit-report-2026-06-03T11-30-00.html ← snapshot após Gate 0
├── audit-report-2026-06-03T14-00-00.html ← snapshot após Fase 1 ... (acumula)
└── lock ← mutex enquanto execute_phase está rodando
ℹ️
Branches Git de fase seguem o padrão jdk-migration/phase-N-YYYYMMDDHHMMSS. Mesmo branches de fases que falharam são preservadas — elas servem como evidência de auditoria e podem ser inspecionadas a qualquer momento com git log --all.
⚠️
O arquivo jdk-migration.config.json contém o estado do state machine da migração. Nunca edite manualmente os campos phases[N].gateToken — tokens inválidos bloqueiam a fase seguinte e precisam ser regenerados via approve_gate.
