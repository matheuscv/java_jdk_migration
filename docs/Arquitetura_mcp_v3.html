Arquitetura em Camadas

Separação de responsabilidades — da surface MCP à knowledge base. A ferramenta orquestra, não reimplementa transformações.

Cada camada tem uma responsabilidade única e se comunica apenas com a camada adjacente. O agente de IA toca somente a MCP Surface; toda a lógica de governança, isolamento Git e validação de versões de JDK vive em camadas inferiores que o agente não consegue subverter.

Controle de JDK — JAVA_HOME · sourceJdkHome · targetJdkHome

Como o MCP separa o JDK de origem do projeto do JDK 21 de destino — sem depender do JAVA_HOME global

O processo MCP normalmente roda com JAVA_HOME apontando para o JDK 21. Depender desse valor para detectar o JDK do projeto-alvo seria um erro: confundiria origem com destino. Por isso o MCP mantém dois conceitos independentes, cada um com regras de validação rígidas.

🟠 sourceJdkHome ORIGEM

JDK da aplicação que será migrada — sempre JDK 6 ou 8, NUNCA 21. Usado na Fase 0 para compilar o projeto antes do jdeprscan, garantindo análise de bytecode completa.

  • Busca casada com a versão do projeto (6 ou 8)
  • Nunca usa JAVA_HOME direto — evita pegar o JDK 21
  • Se um candidato for versão 21, é rejeitado
  • Cadeia: SOURCE_JAVA_HOME → JAVA_HOME_8/JDK_8 → paths comuns → pede ao usuário

🔵 targetJdkHome DESTINO

JDK de destino — sempre e somente JDK 21. Usado pelas fases 1–5 para compilar e validar a aplicação já modernizada.

  • Valida que o candidato é realmente versão 21
  • Qualquer versão ≠ 21 é descartada
  • Cadeia: JAVA_HOME_21/JDK_21 → JAVA_HOME → paths comuns → PATH
  • Se ausente, o MCP solicita o path do JDK 21 ao usuário
🧬
Isolamento por processo filho: ao compilar, o MCP monta um env dedicado para o processo do Maven/Gradle — JAVA_HOME e PATH são reescritos apontando para o JDK correto (origem na Fase 0, destino nas Fases 1–5). O JAVA_HOME do próprio processo MCP nunca é alterado.

Gate de Aprovação em Dois Fatores

PIN humano (fator 1) + token HMAC assinado (fator 2) — impossível de uma automação burlar

Nenhuma fase avança sem aprovação humana real. O MCP impõe isso com dois mecanismos independentes que se complementam: um PIN efêmero que só o humano vê e digita, e um token criptográfico que prova que o gate foi legitimamente aprovado.

📟 Fator 1 — PIN de Aprovação request_gate_approval

Gera um PIN aleatório de 6 dígitos exibido ao responsável técnico. O agente de IA não tem acesso ao arquivo onde o PIN é armazenado.

  • PIN de 6 dígitos via crypto.randomInt
  • Salvo em .jdk-migration/.gate-pins.json — nunca retornado por nenhuma tool
  • Validade de 30 minutos; expira automaticamente
  • Uso único — consumido na validação bem-sucedida
  • Só pode ser gerado se a fase estiver em awaiting_gate

🔏 Fator 2 — Token HMAC approve_gate

Após validar o PIN e o nome do aprovador, emite um token assinado por HMAC-SHA256 derivado do caminho do projeto.

  • Bloqueia nomes de sistemas (bot, claude, ci, ai, gpt…)
  • Assinatura HMAC-SHA256 com segredo derivado do projectPath
  • Validade de 30 dias; timestamp embutido para auditoria
  • Único por fase — exigido pelo execute_phase da fase seguinte
  • timingSafeEqual contra ataques de timing
jdkm.1.{phase}.{issuedAtHex}.{hmac32}
🛡️
Por que dois fatores? O PIN garante presença humana no momento da aprovação (algo que o humano sabe e digita). O token garante integridade e rastreabilidade ao longo do tempo (algo assinado que prova a aprovação). Mesmo que um agente tente chamar approve_gate sozinho, ele não conhece o PIN — que vive em um arquivo fora do seu alcance.

As 11 MCP Tools — Referência Completa

Tudo que o servidor expõe ao agente de IA · ● = novas desde v0.3.x

discover_project
Escaneia a aplicação sem alterá-la. Detecta e valida JDKs (origem/destino), compila com o JDK de origem, roda jdeprscan/jdeps/source-scanner, varre Docker/CI e correlaciona com a knowledge base. Primeiro comando da migração.
Input: projectPath · toolOverrides Sem alteração de código
build_migration_plan
Consolida o relatório de descoberta em plano faseado com gates. Seleciona recipes por stack, estima esforço em dias e persiste migration-plan.json. Requer discover_project anterior.
Input: projectPath Gera migration-plan.json
execute_phase
Aplica uma fase aprovada. Valida o gate token, cria branch jdk-migration/phase-N-YYYYMMDDHHMMSS antes de qualquer alteração, aplica transformações, executa build + testes com o JDK 21, commita e abre PR via syncMigrationBranch (se gh disponível). Suporta dryRun. Rollback automático em falha — branch preservada como evidência.
Exige gate token (fase > 0) Branch jdk-migration/phase-N-<ts> PR automático · Rollback em falha
request_gate_approval NOVO
Gera um PIN de 6 dígitos exibido ao responsável técnico, armazenado fora do alcance da IA. Deve ser chamado antes de approve_gate; o humano fornece o PIN de volta. Validade de 30 minutos.
Fator 1 · PIN Exige fase awaiting_gate
approve_gate
Registra a aprovação humana e emite o token HMAC que libera a fase seguinte. Nunca por automação — rejeita nomes de sistemas e exige o PIN de 6 dígitos. Gera audit report (final na Fase 5).
Human-only Fator 2 · Token
get_phase_status
Retorna o status atual de todas as 6 fases: pending, in_progress, awaiting_gate, approved, completed, failed ou rolled_back — com branches, commits e datas de aprovação. Read-only.
Read-only Sem efeito colateral
rollback_phase
Reverte uma fase aplicada via Git — volta à branch base. A branch de fase é preservada como registro de auditoria. Disponível em in_progress, awaiting_gate ou failed. Não requer token.
Checkout para branch base Branch fase preservada
update_step_status NOVO
Registra o progresso de um step individual (fases A/B/C/D) dentro da fase ativa. Persiste no config para o relatório HTML. Com reportMode=phase-gate-step, gera um novo audit report a cada chamada.
step · owner · phase · status Audit report incremental
record_manual_phase NOVO
Registra uma fase executada manualmente fora do MCP (ex.: execute_phase falhou por EINVAL/ENOENT). Avança para awaiting_gate preservando a trilha, com merge atômico de steps. Não aplica transformações.
Branch isolada obrigatória Merge de steps (upsert)
check_internal_dependencies NOVO
Consulta o registry interno (Nexus 3 / Artifactory) para verificar quais dependências internas têm versões compatíveis com Spring Boot 3. Requer artifactRegistry no config.
Input: projectPath Nexus3 / Artifactory
generate_report
Gera relatório HTML consolidado com trilha de auditoria, registro de riscos, status de fases, progresso de steps e itens manuais. Também disparado automaticamente em fases, gates e steps.
Input: projectPath audit-report-<ts>.html

Rastreio Granular de Steps (A · B · C · D)

Dentro de cada fase, o trabalho é organizado em steps com owner, status e commit — alimentando o relatório

Além do estado macro de cada fase, o MCP rastreia steps individuais via update_step_status e record_manual_phase. Cada step pertence a uma das quatro sub-fases de execução e tem um responsável (claude ou you), status (done/pending/skipped), commit Git e nota opcional.

A
Verificações & Decisões
Pré-condições, análise de impacto e decisões de abordagem antes de tocar o código.
B
Implementação
Aplicação das transformações, edições de código e atualização de dependências.
C
Validação
Build, testes, smoke tests e verificação de paridade funcional.
D
Limpeza
Remoção de código morto, ajustes finais e organização do commit.
⚙️
O campo reportMode controla a granularidade dos relatórios automáticos: phase-gate (padrão) gera report ao fim de cada fase e gate; phase-gate-step também gera um novo snapshot HTML a cada step concluído — útil para acompanhamento em tempo real de migrações longas.

Verificação de Dependências Internas

check_internal_dependencies consulta Nexus 3 / Artifactory por versões compatíveis com Spring Boot 3

Antes da Fase 3 (Jakarta & Frameworks), libs internas da organização podem não ter versão compatível com Spring Boot 3 — o que travaria a migração. A tool check_internal_dependencies consulta o registry configurado e indica, por dependência, se há uma versão provavelmente compatível.

Campo de artifactRegistryDescriçãoExemplo
typeTipo do registry (ou none para desabilitar)nexus3 · artifactory
urlURL base do registry internohttps://nexus.empresa.local
internalGroupIdsPrefixos de groupId considerados internos["com.empresa", "com.empresa.plat"]
📦
A heurística de compatibilidade usa a data de release do Spring Boot 3.0 GA (24/11/2022) como referência: versões publicadas após essa data têm maior probabilidade de suportar SB3. O resultado é um guia de priorização — não substitui a validação funcional.

Princípios Inegociáveis

Regras de governança que o MCP impõe programaticamente

🔐
Gate em Dois Fatores
Nenhuma fase avança sem PIN humano (request_gate_approval) + token HMAC (approve_gate) da fase anterior. PIN fica fora do alcance da IA.
🟠
Source JDK Nunca é 21
A detecção do JDK de origem rejeita qualquer candidato versão 21 e nunca usa JAVA_HOME direto — casa com a versão real do projeto (6/8).
🔵
Target JDK Sempre 21
O JDK de destino é validado como versão 21. Qualquer outra versão é descartada; se ausente, o MCP solicita o path ao usuário.
🛑
EJB: Sem Automação Semântica
Transformações em EJB/JTA nunca são aplicadas automaticamente — o MCP apenas sinaliza e organiza a revisão humana obrigatória.
🌿
Branch Git Isolada por Fase
Cada fase opera em jdk-migration/phase-N-<ts>. Nunca há commit direto na branch principal. Rollback automático em falha.
🧪
dryRun Antes de Stacks Críticas
O parâmetro dryRun: true permite preview do diff sem aplicar mudanças — recomendado antes de fases de alta/crítica complexidade.
📋
Ordem de Recipes Importa
Recipe JDK (UpgradeToJava21) sempre executa antes do recipe WebLogic — ordem oficial Oracle, não pode ser invertida.
👤
approve_gate é Humano-Only
A tool rejeita approverNames que identifiquem sistemas (claude, bot, ci, ai, gpt…). O aprovador deve ser uma pessoa real.
🔁
Trabalho Manual é Auditável
Quando o ambiente quebra, record_manual_phase registra o trabalho feito na CLI em branch isolada, preservando a trilha de auditoria.
🔧
Protocolo Manual: Branch Antes do Trabalho
Em execuções manuais, a branch jdk-migration/phase-N-YYYYMMDDHHMMSS deve ser criada antes de qualquer alteração (git checkout -b). Sem isso, o trabalho fica na branch errada e o rollback independente da fase fica inviável.

Artefatos Gerados no Projeto-Alvo

Estrutura criada pelo MCP na raiz da aplicação em migração

projeto-alvo/
├── jdk-migration.config.json ← state machine + config (fases, JDKs, steps, registry)
└── .jdk-migration/
├── discovery-report.json ← resultado do discover_project (riscos + detectedTools)
├── migration-plan.json ← plano faseado do build_migration_plan
├── audit-report-2026-06-06T10-00-00.html ← snapshot após Fase 0
├── audit-report-2026-06-06T11-30-00.html ← snapshot após Gate 0 ... (acumula)
├── audit-report-final.html ← relatório final fixo (gerado no Gate 5)
├── .gate-pins.json ← PINs de aprovação — FORA do alcance da IA
└── lock ← mutex enquanto execute_phase está rodando
🌿
Branches Git de fase seguem o padrão jdk-migration/phase-N-YYYYMMDDHHMMSS. Mesmo branches de fases que falharam são preservadas — servem como evidência de auditoria e podem ser inspecionadas com git log --all.
⚠️
O jdk-migration.config.json contém o estado do state machine. Nunca edite manualmente os campos phases[N].gateToken — tokens inválidos bloqueiam a fase seguinte e precisam ser regenerados via o fluxo request_gate_approval → approve_gate.
🔒
O arquivo .gate-pins.json nunca é retornado por nenhuma MCP tool — é deliberadamente inacessível ao agente de IA. É o que garante que o fator humano do gate não possa ser contornado por automação.

Isolamento de Branches Git por Fase

Cada fase opera em branch própria — rastreabilidade completa e rollback independente sem tocar na branch principal

O isolamento por branch é um princípio arquitetural imposto pelo execute_phase e pelo protocolo de execução manual. Nenhum commit vai direto na branch principal durante a migração — o trabalho de cada fase tem sua própria trilha no Git, auditável via git log --all a qualquer momento.

⚡ Modo automático — execute_phase

A branch é criada automaticamente pelo MCP antes de qualquer transformação.

  • Padrão: jdk-migration/phase-N-YYYYMMDDHHMMSS
  • Transformações aplicadas exclusivamente nessa branch
  • Commit registrado após build verde com JDK 21
  • PR criado via syncMigrationBranch + gh pr create (se gh disponível)
  • Rollback automático: git checkout <baseBranch> em falha
  • Branch da fase preservada mesmo em rollback — evidência de auditoria

🔧 Modo manual — record_manual_phase

Quando execute_phase falha por problema ambiental (EINVAL, ENOENT, timeout de Maven), o trabalho é feito na CLI.

  • Branch deve ser criada antes de qualquer alteração
  • git checkout -b jdk-migration/phase-N-YYYYMMDDHHMMSS
  • Mesmo padrão de nomenclatura — relatório mostra a branch correta
  • Commit manual na branch criada
  • Registrar com record_manual_phase(branch, commit, note)
  • MCP avança status para awaiting_gate

🔀 syncMigrationBranch — PR automático após cada fase

1. Build verde
execute_phase conclui com sucesso. Commit na branch jdk-migration/phase-N-<ts>
2. syncMigrationBranch
Push da branch para o remote e chamada a createPullRequest via gh pr create
3. Status awaiting_gate
PR aberto no GitHub. Audit report gerado. Gate PIN solicitado ao aprovador humano.
⚠️
Sem branch antes → trilha quebrada. Se o trabalho manual é feito na branch de uma fase anterior, o relatório mostra a branch errada, o PR aponta para o diff incorreto e o rollback_phase da nova fase fica inviável de forma independente. Por isso a branch deve existir antes do primeiro commit.
